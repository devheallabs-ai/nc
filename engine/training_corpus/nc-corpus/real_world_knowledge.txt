--------------------------------------------------------------------------------
1.1 REST APIs and HTTP
--------------------------------------------------------------------------------

REST stands for Representational State Transfer, an architectural style for designing networked applications.
A REST API exposes resources as URLs and uses standard HTTP methods to operate on them.
The base URL of a REST API is often called the endpoint, such as https://api.example.com/v1/users.

HTTP GET retrieves a resource without modifying it and should be idempotent.
HTTP POST creates a new resource on the server and typically returns a 201 Created status.
HTTP PUT replaces an entire resource at the specified URL with the provided payload.
HTTP PATCH applies a partial update to an existing resource, modifying only specified fields.
HTTP DELETE removes a resource from the server and typically returns 204 No Content on success.
HTTP OPTIONS returns the allowed methods and headers for a given endpoint, used in CORS preflight requests.
HTTP HEAD is identical to GET but returns only headers, not the response body.

Idempotent methods produce the same result regardless of how many times they are called.
GET, PUT, DELETE, and HEAD are idempotent methods.
POST is not idempotent because calling it multiple times may create multiple resources.

HTTP status code 200 means OK, the request succeeded.
HTTP status code 201 means Created, a new resource was successfully created.
HTTP status code 204 means No Content, the request succeeded but there is no body to return.
HTTP status code 301 means Moved Permanently, the resource has a new permanent URL.
HTTP status code 302 means Found, the resource is temporarily at a different URL.
HTTP status code 304 means Not Modified, the cached version is still valid.
HTTP status code 400 means Bad Request, the server cannot process the request due to client error.
HTTP status code 401 means Unauthorized, the client must authenticate before accessing the resource.
HTTP status code 403 means Forbidden, the client is authenticated but lacks permission.
HTTP status code 404 means Not Found, the requested resource does not exist.
HTTP status code 405 means Method Not Allowed, the HTTP method is not supported for this endpoint.
HTTP status code 409 means Conflict, the request conflicts with the current state of the resource.
HTTP status code 422 means Unprocessable Entity, the request syntax is valid but semantically incorrect.
HTTP status code 429 means Too Many Requests, the client has exceeded the rate limit.
HTTP status code 500 means Internal Server Error, the server encountered an unexpected condition.
HTTP status code 502 means Bad Gateway, the server received an invalid response from an upstream server.
HTTP status code 503 means Service Unavailable, the server is temporarily unable to handle requests.
HTTP status code 504 means Gateway Timeout, the upstream server did not respond in time.

The Content-Type header specifies the media type of the request or response body.
Application/json is the most common Content-Type for REST APIs.
The Accept header tells the server which media types the client can handle.
The Authorization header carries credentials such as Bearer tokens or Basic authentication.
The Cache-Control header directs caching mechanisms on how to cache the response.
The ETag header provides a version identifier for a resource, enabling conditional requests.
The If-None-Match header sends a previously received ETag to check if the resource has changed.
The X-Request-ID header carries a unique identifier for tracing a request across services.
The Content-Length header indicates the size of the response body in bytes.
The Location header in a 201 response contains the URL of the newly created resource.

CORS stands for Cross-Origin Resource Sharing, a mechanism that allows restricted resources to be requested from another domain.
A preflight request is an OPTIONS request sent by the browser before the actual cross-origin request.
The Access-Control-Allow-Origin header specifies which origins may access the resource.
The Access-Control-Allow-Methods header lists the HTTP methods permitted in cross-origin requests.
The Access-Control-Allow-Headers header lists the headers that can be used in cross-origin requests.

REST API versioning can be done via URL path (v1/users), query parameter (users?version=1), or header (Accept: application/vnd.api+json;version=1).
HATEOAS stands for Hypermedia as the Engine of Application State, where API responses include links to related actions.
Pagination in REST APIs is typically implemented using query parameters like page and limit or cursor-based tokens.
Rate limiting protects APIs from abuse by restricting the number of requests a client can make in a time window.
The X-RateLimit-Limit header indicates the maximum number of requests allowed in a window.
The X-RateLimit-Remaining header shows how many requests remain in the current window.
The Retry-After header tells the client how many seconds to wait before retrying after a 429 response.

API authentication methods include API keys, OAuth 2.0 tokens, JWT bearer tokens, and mutual TLS.
An API key is a simple string passed in a header or query parameter to identify the calling application.
Bearer tokens are passed in the Authorization header as "Bearer <token>" and grant access to protected resources.
Webhook endpoints allow servers to push real-time event notifications to client-registered URLs.

1.2 Database Concepts
--------------------------------------------------------------------------------

A relational database stores data in tables with rows and columns, enforcing a predefined schema.
SQL stands for Structured Query Language and is used to manage and query relational databases.
Common relational databases include PostgreSQL, MySQL, MariaDB, Oracle, and Microsoft SQL Server.
PostgreSQL is an advanced open-source relational database known for extensibility and standards compliance.
MySQL is one of the most widely deployed open-source relational databases, popular for web applications.

A NoSQL database provides flexible schemas and is designed for specific data models like documents, key-value pairs, graphs, or wide columns.
Document databases like MongoDB store data as JSON-like documents, allowing nested structures.
Key-value stores like Redis and DynamoDB map keys to values for extremely fast lookups.
Wide-column stores like Cassandra and HBase organize data into column families for high write throughput.
Graph databases like Neo4j store data as nodes and edges, optimized for relationship-heavy queries.

A primary key uniquely identifies each row in a table and cannot contain NULL values.
A foreign key is a column that references the primary key of another table, establishing a relationship.
A composite key is a primary key made up of two or more columns together.
An index is a data structure that improves the speed of data retrieval operations on a table.
A B-tree index is the most common index type, efficient for equality and range queries.
A hash index provides constant-time lookups for exact-match queries but does not support range queries.
A covering index includes all columns needed by a query, eliminating the need to access the main table.
A partial index only indexes rows that satisfy a specified condition, reducing index size.
A full-text index enables efficient searching of text content within columns.
Over-indexing slows down write operations because every insert, update, or delete must also update the indexes.

Normalization is the process of organizing database tables to reduce redundancy and dependency.
First normal form (1NF) requires that each column contains only atomic (indivisible) values.
Second normal form (2NF) requires 1NF plus all non-key columns must depend on the entire primary key.
Third normal form (3NF) requires 2NF plus no non-key column depends on another non-key column.
Denormalization intentionally introduces redundancy to improve read performance at the cost of write complexity.

ACID properties ensure reliable database transactions.
Atomicity means a transaction is all-or-nothing; either all operations succeed or none do.
Consistency means a transaction brings the database from one valid state to another.
Isolation means concurrent transactions do not interfere with each other.
Durability means once a transaction is committed, it persists even after a system crash.

A transaction is a sequence of database operations treated as a single logical unit of work.
The COMMIT statement finalizes a transaction, making all changes permanent.
The ROLLBACK statement undoes all changes made during the current transaction.
A savepoint allows you to roll back to a specific point within a transaction without aborting the entire transaction.
A deadlock occurs when two transactions each hold a lock the other needs, causing both to wait indefinitely.
Optimistic locking checks for conflicts at commit time, assuming conflicts are rare.
Pessimistic locking acquires locks before making changes, preventing concurrent access.

The CAP theorem states that a distributed system can provide at most two of three guarantees: Consistency, Availability, and Partition Tolerance.
Eventual consistency means all replicas will converge to the same state given enough time without new updates.
Strong consistency means every read receives the most recent write or an error.

A join combines rows from two or more tables based on a related column.
An INNER JOIN returns only rows where there is a match in both tables.
A LEFT JOIN returns all rows from the left table and matching rows from the right table.
A RIGHT JOIN returns all rows from the right table and matching rows from the left table.
A FULL OUTER JOIN returns all rows from both tables, with NULLs where there is no match.
A CROSS JOIN returns the Cartesian product of both tables.

A stored procedure is a precompiled collection of SQL statements stored in the database.
A trigger is a stored procedure that automatically executes in response to certain events on a table.
A view is a virtual table defined by a SQL query, presenting data from one or more underlying tables.
A materialized view stores the result of a query physically on disk and must be refreshed to reflect changes.

Database sharding distributes data across multiple servers by splitting rows based on a shard key.
Horizontal partitioning splits a table into multiple tables with the same columns but different rows.
Vertical partitioning splits a table by columns, placing frequently accessed columns in one partition.
A read replica is a copy of the primary database that handles read queries to reduce load on the primary.
Connection pooling maintains a cache of database connections that can be reused, reducing connection overhead.

1.3 Microservices Architecture
--------------------------------------------------------------------------------

Microservices architecture decomposes an application into small, independently deployable services.
Each microservice owns its own data store, avoiding shared databases between services.
Services communicate via lightweight protocols like HTTP/REST, gRPC, or asynchronous messaging.
A monolithic architecture bundles all functionality into a single deployable unit.
The strangler fig pattern gradually replaces parts of a monolith with microservices over time.

Service discovery is the mechanism by which services locate each other on the network.
Client-side discovery means the client queries a service registry and selects an instance.
Server-side discovery means a load balancer queries the service registry on behalf of the client.
Consul, Eureka, and etcd are popular service discovery tools.
A service registry is a database of available service instances and their network locations.
Services register themselves on startup and deregister on shutdown.
Health checks verify that registered service instances are still alive and able to handle requests.

A load balancer distributes incoming requests across multiple service instances.
Round-robin load balancing sends each request to the next server in sequence.
Least-connections load balancing routes requests to the server with the fewest active connections.
Weighted load balancing assigns more traffic to servers with higher capacity.
Layer 4 load balancing operates at the transport level, routing based on IP and port.
Layer 7 load balancing operates at the application level, routing based on HTTP headers, URLs, or cookies.
NGINX, HAProxy, and Envoy are widely used load balancers and reverse proxies.

A circuit breaker prevents cascading failures by stopping requests to a failing service.
The circuit breaker has three states: Closed (normal), Open (failing), and Half-Open (testing recovery).
In the Closed state, requests flow normally and failures are counted.
When the failure threshold is exceeded, the circuit breaker transitions to Open, rejecting all requests immediately.
After a timeout period, the circuit breaker enters Half-Open, allowing a limited number of test requests.
If test requests succeed, the circuit returns to Closed; if they fail, it returns to Open.
Bulkheads isolate failures by partitioning resources so that a failure in one part does not affect others.
Retry with exponential backoff gradually increases the delay between retry attempts.
Jitter adds randomness to backoff delays to prevent thundering herd problems.

An API gateway is a single entry point for all client requests to backend microservices.
The API gateway handles cross-cutting concerns like authentication, rate limiting, and request routing.
Kong, Apigee, and AWS API Gateway are popular API gateway products.
Backend-for-Frontend (BFF) is a pattern where each frontend type has its own backend gateway.

Event-driven architecture uses events to trigger and communicate between services.
A message broker facilitates asynchronous communication between producers and consumers.
Apache Kafka is a distributed event streaming platform used for high-throughput messaging.
RabbitMQ is a message broker that implements the Advanced Message Queuing Protocol (AMQP).
The Saga pattern manages distributed transactions across microservices using a sequence of local transactions.
Choreography-based sagas have each service publish events that trigger the next step.
Orchestration-based sagas use a central coordinator to manage the transaction steps.
The Outbox pattern ensures reliable event publishing by writing events to a database table before dispatching.
Idempotency ensures that processing the same message multiple times produces the same result.
A dead letter queue stores messages that could not be processed after multiple retry attempts.

A sidecar proxy runs alongside a service instance and handles networking concerns like TLS, retries, and observability.
A service mesh manages service-to-service communication using sidecar proxies.
Istio and Linkerd are popular service mesh implementations.
Distributed tracing tracks requests as they flow across multiple services.
A correlation ID is a unique identifier attached to a request that propagates across all services for tracing.
Jaeger and Zipkin are popular distributed tracing systems.
Centralized logging aggregates logs from all services into a single searchable system.
The ELK stack (Elasticsearch, Logstash, Kibana) is a common centralized logging solution.

1.4 DevOps
--------------------------------------------------------------------------------

DevOps is a set of practices that combines software development and IT operations to shorten the development lifecycle.
Continuous Integration (CI) is the practice of merging code changes into a shared repository frequently, with automated builds and tests.
Continuous Delivery (CD) ensures that code is always in a deployable state by automating the release process.
Continuous Deployment automatically deploys every change that passes the pipeline to production.
A CI/CD pipeline is an automated workflow that builds, tests, and deploys code changes.
Pipeline stages typically include: source, build, test, staging, and production deployment.
Jenkins, GitHub Actions, GitLab CI, CircleCI, and Travis CI are popular CI/CD tools.

Infrastructure as Code (IaC) manages infrastructure through machine-readable definition files.
Terraform is a popular IaC tool that provisions infrastructure across multiple cloud providers.
Terraform uses a declarative configuration language called HCL (HashiCorp Configuration Language).
Terraform state files track the current state of managed infrastructure and should be stored remotely.
Ansible is an agentless configuration management tool that uses YAML playbooks.
Pulumi allows writing infrastructure definitions in general-purpose programming languages.
CloudFormation is AWS's native IaC service for provisioning AWS resources.

Docker is a platform for building, shipping, and running applications in containers.
A container is a lightweight, standalone, executable package that includes everything needed to run an application.
A Docker image is a read-only template used to create containers.
A Dockerfile contains instructions to build a Docker image, typically starting with a FROM directive.
Multi-stage Docker builds use multiple FROM statements to create smaller final images.
Docker Compose defines and runs multi-container applications using a YAML configuration file.
A container registry stores and distributes Docker images, such as Docker Hub or Amazon ECR.
Container images should be kept small by using minimal base images like Alpine Linux.
Each instruction in a Dockerfile creates a new layer; layers are cached to speed up subsequent builds.

Kubernetes (K8s) is an open-source container orchestration platform for automating deployment, scaling, and management.
A Kubernetes Pod is the smallest deployable unit, consisting of one or more containers that share networking and storage.
A Deployment manages a set of identical Pods and handles rolling updates and rollbacks.
A ReplicaSet ensures that a specified number of Pod replicas are running at any given time.
A Kubernetes Service provides a stable network endpoint for accessing a set of Pods.
ClusterIP is the default Service type, accessible only within the cluster.
NodePort exposes a Service on a static port on each node's IP address.
LoadBalancer provisions an external load balancer to expose a Service to the internet.
An Ingress resource manages external HTTP and HTTPS access to Services within the cluster.
A ConfigMap stores non-confidential configuration data as key-value pairs.
A Secret stores sensitive data like passwords and tokens in a base64-encoded format.
A PersistentVolume provides storage that outlives the Pod that uses it.
A PersistentVolumeClaim is a request for storage by a Pod.
A Namespace provides a mechanism for isolating groups of resources within a cluster.
Horizontal Pod Autoscaler automatically scales the number of Pods based on CPU utilization or custom metrics.
A DaemonSet ensures that a copy of a Pod runs on every node in the cluster.
A StatefulSet manages Pods that require stable identities and persistent storage.
Helm is a package manager for Kubernetes that uses charts to define, install, and upgrade applications.
Kustomize is a Kubernetes-native tool for customizing YAML configurations without templates.

Monitoring tracks the state and performance of systems using metrics, logs, and traces.
Prometheus is an open-source monitoring system that collects metrics via a pull model.
Grafana is a visualization platform commonly used to create dashboards for Prometheus metrics.
Alertmanager routes alerts from Prometheus to notification channels like Slack, email, or PagerDuty.
The four golden signals of monitoring are latency, traffic, errors, and saturation.
SLI (Service Level Indicator) is a quantitative measure of service behavior, such as request latency.
SLO (Service Level Objective) is a target value for an SLI, such as 99.9% of requests under 200ms.
SLA (Service Level Agreement) is a contract with consequences if SLOs are not met.
An error budget is the maximum allowable amount of unreliability within an SLO period.

Blue-green deployment maintains two identical production environments and switches traffic between them.
Canary deployment gradually routes a small percentage of traffic to the new version before full rollout.
Rolling deployment incrementally replaces old instances with new ones, maintaining availability throughout.
Feature flags allow toggling features on or off at runtime without deploying new code.
GitOps uses Git as the single source of truth for declarative infrastructure and application configuration.
ArgoCD and Flux are popular GitOps tools for Kubernetes.

1.5 Security
--------------------------------------------------------------------------------

The OWASP Top 10 is a list of the most critical web application security risks.
Injection attacks occur when untrusted data is sent to an interpreter as part of a command or query.
SQL injection inserts malicious SQL code through user input to manipulate database queries.
Parameterized queries (prepared statements) prevent SQL injection by separating code from data.
Cross-Site Scripting (XSS) injects malicious scripts into web pages viewed by other users.
Stored XSS saves the malicious script on the server, affecting all users who view the infected page.
Reflected XSS includes the malicious script in the URL, affecting only users who click the link.
DOM-based XSS manipulates the client-side DOM without server involvement.
Content Security Policy (CSP) headers restrict which scripts and resources a page can load, mitigating XSS.
Cross-Site Request Forgery (CSRF) tricks an authenticated user into performing unintended actions.
CSRF tokens are unique, unpredictable values that verify requests originate from the legitimate application.
Broken Authentication occurs when authentication mechanisms are implemented incorrectly.
Session fixation attacks force a user to use a known session ID controlled by the attacker.
Insecure Direct Object References allow attackers to access unauthorized resources by modifying identifiers.
Security Misconfiguration includes default credentials, unnecessary features, and verbose error messages.
Sensitive Data Exposure occurs when applications do not adequately protect data like passwords or credit cards.
XML External Entities (XXE) attacks exploit XML parsers to read files, perform SSRF, or cause denial of service.
Broken Access Control allows users to act outside their intended permissions.
Server-Side Request Forgery (SSRF) tricks the server into making requests to unintended internal resources.

JSON Web Token (JWT) is a compact, URL-safe token format for securely transmitting claims between parties.
A JWT consists of three parts: Header, Payload, and Signature, separated by dots.
The Header specifies the signing algorithm (e.g., HS256, RS256) and token type.
The Payload contains claims such as subject (sub), expiration time (exp), and issued at (iat).
The Signature verifies that the token has not been tampered with.
HS256 uses a shared secret key for signing and verification (symmetric).
RS256 uses a private key for signing and a public key for verification (asymmetric).
JWTs should have short expiration times and be refreshed using refresh tokens.
Refresh tokens are long-lived tokens stored securely that can be exchanged for new access tokens.
Never store sensitive information in the JWT payload because it is only base64-encoded, not encrypted.
JWE (JSON Web Encryption) provides encryption for JWT payloads when confidentiality is needed.

OAuth 2.0 is an authorization framework that enables third-party applications to access resources on behalf of a user.
The Authorization Code flow is the most secure OAuth 2.0 flow for server-side applications.
The Authorization Code flow with PKCE (Proof Key for Code Exchange) is recommended for single-page and mobile applications.
The Client Credentials flow is used for machine-to-machine communication without user involvement.
The Implicit flow is deprecated due to security concerns around exposing tokens in the URL.
An access token grants temporary permission to access specific resources.
A refresh token allows obtaining new access tokens without requiring user re-authentication.
Scopes define the specific permissions an access token grants.
OpenID Connect (OIDC) is an identity layer built on top of OAuth 2.0 for authentication.
The ID token in OIDC contains claims about the user's identity.

Symmetric encryption uses the same key for both encryption and decryption.
AES (Advanced Encryption Standard) is the most widely used symmetric encryption algorithm.
AES-256 uses a 256-bit key and is considered secure against brute-force attacks.
Asymmetric encryption uses a pair of keys: a public key for encryption and a private key for decryption.
RSA is a widely used asymmetric encryption algorithm, though it is slower than symmetric algorithms.
Elliptic Curve Cryptography (ECC) provides equivalent security to RSA with shorter key lengths.
A hash function converts input of any size into a fixed-size output that cannot be reversed.
SHA-256 produces a 256-bit hash and is widely used for data integrity verification.
Bcrypt is a password hashing function that includes a salt and a configurable work factor.
Argon2 is the winner of the Password Hashing Competition and is recommended for new applications.
A salt is a random value added to a password before hashing to prevent rainbow table attacks.
HTTPS encrypts HTTP traffic using TLS (Transport Layer Security).
TLS 1.3 is the latest version, providing improved security and performance over TLS 1.2.
A TLS certificate is issued by a Certificate Authority (CA) to verify the identity of a website.
Let's Encrypt is a free, automated Certificate Authority that provides TLS certificates.
Certificate pinning ensures that a client only accepts specific certificates for a domain.
The principle of least privilege grants users only the minimum permissions necessary for their tasks.

1.6 Testing
--------------------------------------------------------------------------------

A unit test verifies the behavior of a single function or method in isolation from external dependencies.
Unit tests should be fast, independent, and deterministic.
A mock is a simulated object that mimics the behavior of a real dependency in a controlled way.
A stub provides predefined responses to method calls during testing.
A spy records information about how it was called, allowing verification of interactions.
Test doubles is the general term for mocks, stubs, spies, fakes, and dummies.

Integration tests verify that multiple components work together correctly.
Integration tests often involve real databases, file systems, or network services.
Contract testing verifies that services communicate correctly according to agreed-upon interfaces.
Pact is a popular tool for consumer-driven contract testing.

End-to-end (E2E) tests simulate real user scenarios by testing the entire application stack.
E2E tests are slower and more fragile than unit or integration tests.
Cypress, Playwright, and Selenium are popular E2E testing frameworks for web applications.
The testing pyramid recommends many unit tests, fewer integration tests, and even fewer E2E tests.
The testing trophy (an alternative to the pyramid) emphasizes integration tests as the most valuable layer.

Test-Driven Development (TDD) writes tests before writing the production code.
The TDD cycle is Red-Green-Refactor: write a failing test, make it pass, then improve the code.
Behavior-Driven Development (BDD) extends TDD by writing tests in natural language using Given-When-Then format.
Acceptance Test-Driven Development (ATDD) involves stakeholders in defining acceptance criteria before coding.

Code coverage measures the percentage of code executed during testing.
Line coverage measures whether each line of code was executed.
Branch coverage measures whether each branch of conditional logic was taken.
High code coverage does not guarantee the absence of bugs; it only ensures code was exercised.
Mutation testing introduces small changes to the code and checks whether tests catch them.

Performance testing measures how a system behaves under load.
Load testing applies expected production-level traffic to identify performance bottlenecks.
Stress testing pushes the system beyond normal capacity to find its breaking point.
Soak testing runs the system under sustained load for an extended period to detect memory leaks.
JMeter, Gatling, and k6 are popular performance testing tools.
Chaos engineering intentionally introduces failures to test system resilience.
Netflix's Chaos Monkey randomly terminates production instances to ensure fault tolerance.

Property-based testing generates random inputs to verify that properties hold for all values.
Snapshot testing captures the output of a component and compares it against a stored reference.
Regression testing ensures that new changes do not break existing functionality.
Smoke testing performs basic checks to verify that the most critical functions work after deployment.
A test fixture sets up the preconditions required for a test to run.

1.7 Git Workflows
--------------------------------------------------------------------------------

Git is a distributed version control system that tracks changes to files over time.
A Git repository contains the complete history of a project and all its branches.
The working directory contains the current state of your files.
The staging area (index) holds changes that will be included in the next commit.
A commit is a snapshot of the project at a point in time, identified by a SHA-1 hash.

A branch is a lightweight, movable pointer to a commit.
The main (or master) branch typically represents the production-ready state of the code.
A feature branch is created from main to develop a specific feature in isolation.
The "git checkout -b feature-name" command creates and switches to a new branch.

Git merge combines the changes from one branch into another.
A fast-forward merge moves the branch pointer forward when there is no divergence.
A three-way merge creates a new merge commit when both branches have diverged.
Merge conflicts occur when the same lines are changed in both branches.
Resolving a merge conflict requires manually editing the conflicting files and committing the result.

Git rebase moves a branch's commits on top of another branch, creating a linear history.
Interactive rebase (git rebase -i) allows squashing, reordering, and editing commits.
Rebasing rewrites commit history and should not be used on shared branches.
Squashing combines multiple commits into a single commit for a cleaner history.

Gitflow uses dedicated branches for features, releases, hotfixes, develop, and main.
In Gitflow, the develop branch serves as the integration branch for features.
Release branches are created from develop and merged into both main and develop.
Hotfix branches are created from main to fix critical production issues.

GitHub Flow is a simpler workflow: create a branch, make changes, open a pull request, merge to main.
Trunk-based development has all developers commit to a single branch with short-lived feature branches.

A pull request (or merge request) is a proposal to merge changes from one branch into another.
Code review in pull requests helps catch bugs, enforce standards, and share knowledge.
A pull request should be small and focused on a single change for easier review.

Git tags mark specific points in history, typically used for release versions.
A lightweight tag is simply a pointer to a commit.
An annotated tag stores additional metadata including the tagger name, date, and message.
Semantic versioning uses the format MAJOR.MINOR.PATCH to communicate the nature of changes.

Cherry-picking applies a specific commit from one branch to another.
Git stash temporarily saves uncommitted changes so you can switch branches.
Git bisect uses binary search to find the commit that introduced a bug.
A .gitignore file specifies patterns of files that Git should not track.
Git hooks are scripts that run automatically at certain points in the Git workflow.
Pre-commit hooks run before a commit is created, commonly used for linting and formatting.

1.8 Cloud Concepts
--------------------------------------------------------------------------------

Cloud computing provides on-demand computing resources over the internet.
IaaS (Infrastructure as a Service) provides virtual machines, storage, and networking.
PaaS (Platform as a Service) provides a platform for building and deploying applications without managing infrastructure.
SaaS (Software as a Service) delivers fully managed applications over the internet.
FaaS (Function as a Service) runs individual functions in response to events, also known as serverless.

Amazon Web Services (AWS) is the largest cloud provider by market share.
AWS EC2 provides resizable virtual machines in the cloud.
AWS S3 provides scalable object storage with high durability.
AWS Lambda is a serverless compute service that runs code in response to events.
AWS RDS is a managed relational database service supporting multiple database engines.
AWS DynamoDB is a fully managed NoSQL key-value and document database.
AWS CloudFront is a content delivery network for low-latency content distribution.
AWS SQS is a fully managed message queue service.
AWS SNS is a fully managed publish-subscribe messaging service.
AWS EKS is a managed Kubernetes service.
AWS IAM manages access to AWS services and resources.

Google Cloud Platform (GCP) is a suite of cloud computing services by Google.
GCP Compute Engine provides virtual machines similar to AWS EC2.
GCP Cloud Storage provides object storage similar to AWS S3.
GCP Cloud Functions is a serverless execution environment similar to AWS Lambda.
GCP BigQuery is a fully managed, serverless data warehouse for analytics.
GCP Cloud Run runs containers in a fully managed serverless environment.
GKE (Google Kubernetes Engine) is a managed Kubernetes service on GCP.

Microsoft Azure is a cloud platform by Microsoft, deeply integrated with enterprise tools.
Azure Virtual Machines provides compute resources similar to AWS EC2.
Azure Blob Storage provides object storage similar to AWS S3.
Azure Functions is a serverless compute service similar to AWS Lambda.
Azure Cosmos DB is a globally distributed, multi-model database service.
AKS (Azure Kubernetes Service) is a managed Kubernetes service on Azure.

Serverless computing automatically manages the server infrastructure, scaling from zero to handle any load.
Cold starts occur when a serverless function is invoked after a period of inactivity, causing latency.
Provisioned concurrency pre-warms serverless functions to eliminate cold start latency.
Serverless functions are typically limited by execution duration, memory, and payload size.
Serverless architectures are event-driven, triggered by HTTP requests, database changes, file uploads, or schedules.

A Content Delivery Network (CDN) caches content at edge locations closer to users for lower latency.
Edge computing processes data near the source of data generation rather than in a centralized data center.
A region is a geographic area containing one or more data centers.
An availability zone is an isolated data center within a region with independent power and networking.
Multi-region deployment distributes an application across multiple geographic regions for lower latency and resilience.
Auto-scaling automatically adjusts the number of compute resources based on demand.
Spot instances (AWS) or preemptible VMs (GCP) offer unused capacity at steep discounts but can be reclaimed.
Reserved instances provide significant discounts in exchange for committing to a usage level for one or three years.
Object storage is designed for unstructured data like images, videos, and backups, accessed via HTTP.
Block storage provides raw storage volumes that can be attached to virtual machines, similar to physical hard drives.

2.1 Neural Networks
--------------------------------------------------------------------------------

An artificial neural network is a computational model inspired by biological neural networks.
A neuron (or node) is the basic unit of a neural network that receives inputs, applies weights, and produces an output.
Each input to a neuron is multiplied by a weight that determines its importance.
A bias is an additional parameter added to the weighted sum before applying the activation function.
The weighted sum is computed as: z = w1*x1 + w2*x2 + ... + wn*xn + bias.
An activation function introduces non-linearity, allowing the network to learn complex patterns.
The sigmoid activation function outputs values between 0 and 1, computed as 1 / (1 + e^(-z)).
The ReLU (Rectified Linear Unit) activation outputs max(0, z), which helps avoid the vanishing gradient problem.
Leaky ReLU outputs z if z > 0, otherwise alpha * z, where alpha is a small positive constant.
The tanh activation outputs values between -1 and 1, often used in hidden layers.
The softmax activation converts a vector of values into a probability distribution, commonly used in the output layer for classification.
GELU (Gaussian Error Linear Unit) is used in modern transformer architectures.

A feedforward neural network passes information in one direction, from input to output.
The input layer receives the raw data features.
Hidden layers perform intermediate computations between the input and output layers.
The output layer produces the final prediction or classification.
A deep neural network has multiple hidden layers, enabling it to learn hierarchical representations.
The universal approximation theorem states that a neural network with one hidden layer can approximate any continuous function.

Backpropagation is the algorithm used to compute gradients for training neural networks.
Backpropagation applies the chain rule of calculus to propagate the error backward through the network.
The loss function measures the difference between the network's prediction and the actual target.
Mean Squared Error (MSE) is a common loss function for regression tasks.
Cross-entropy loss is the standard loss function for classification tasks.
Gradient descent updates weights in the direction that minimizes the loss function.
The gradient is the vector of partial derivatives of the loss with respect to each weight.
Stochastic Gradient Descent (SGD) updates weights using one training example at a time.
Mini-batch gradient descent updates weights using a small batch of training examples.
Adam optimizer combines momentum and adaptive learning rates for efficient training.
AdamW adds weight decay regularization to the Adam optimizer.
The vanishing gradient problem occurs when gradients become extremely small in deep networks, slowing learning.
The exploding gradient problem occurs when gradients become extremely large, causing unstable updates.
Gradient clipping limits the maximum gradient value to prevent exploding gradients.
Batch normalization normalizes layer inputs to stabilize and accelerate training.
Layer normalization normalizes across features within each sample, commonly used in transformers.
Residual connections (skip connections) add the input of a layer to its output, helping gradients flow through deep networks.
Dropout randomly sets a fraction of neurons to zero during training to prevent overfitting.

A Convolutional Neural Network (CNN) uses convolutional layers to extract spatial features from data.
A convolutional layer applies learnable filters (kernels) that slide across the input to detect patterns.
Pooling layers reduce spatial dimensions, with max pooling and average pooling being most common.
CNNs are primarily used for image classification, object detection, and image segmentation.

A Recurrent Neural Network (RNN) processes sequential data by maintaining a hidden state.
Long Short-Term Memory (LSTM) networks use gating mechanisms to capture long-range dependencies.
Gated Recurrent Units (GRU) are a simplified version of LSTM with fewer parameters.
RNNs suffer from difficulty learning long-range dependencies due to vanishing gradients.

An autoencoder compresses input into a lower-dimensional latent representation and reconstructs it.
A variational autoencoder (VAE) generates new data by sampling from a learned latent distribution.
Generative Adversarial Networks (GANs) use two competing networks: a generator and a discriminator.

--------------------------------------------------------------------------------
2.2 Training
--------------------------------------------------------------------------------

An epoch is one complete pass through the entire training dataset.
A batch is a subset of the training data processed together in one forward and backward pass.
Batch size is the number of training examples in one batch; common values are 16, 32, 64, 128, and 256.
Larger batch sizes use more memory but provide more stable gradient estimates.
Smaller batch sizes introduce more noise, which can help escape local minima.

The learning rate controls how much the weights are updated in each step.
A high learning rate may cause the model to overshoot the optimal weights and diverge.
A low learning rate causes slow convergence and may get stuck in local minima.
Learning rate scheduling reduces the learning rate over time, allowing finer adjustments as training progresses.
Cosine annealing gradually decreases the learning rate following a cosine curve.
Warmup gradually increases the learning rate from a small value at the beginning of training.
One-cycle policy uses a learning rate that increases then decreases over one training cycle.

Overfitting occurs when a model learns noise in the training data and performs poorly on new data.
Underfitting occurs when a model is too simple to capture the underlying patterns in the data.
The bias-variance tradeoff describes the tension between underfitting (high bias) and overfitting (high variance).
Regularization techniques reduce overfitting by constraining the model.
L1 regularization adds the absolute value of weights to the loss, encouraging sparsity.
L2 regularization (weight decay) adds the squared weights to the loss, penalizing large weights.
Early stopping halts training when validation performance starts to degrade.
Data augmentation increases training data by applying transformations like rotation, flipping, and cropping.
Cross-validation partitions data into folds for more robust model evaluation.
K-fold cross-validation trains and evaluates the model k times, each time using a different fold as validation.
The training set is used to update model weights.
The validation set is used to tune hyperparameters and monitor for overfitting.
The test set is used only for final evaluation and should never influence model development.
Hyperparameter tuning searches for the optimal configuration of learning rate, batch size, architecture, and other settings.
Grid search exhaustively tries all combinations of hyperparameters.
Random search samples hyperparameter combinations randomly and is often more efficient than grid search.
Bayesian optimization builds a probabilistic model to guide the hyperparameter search.

Transfer learning uses a model pre-trained on one task as a starting point for a different but related task.
Fine-tuning adjusts all or some layers of a pre-trained model on a new dataset.
Feature extraction freezes the pre-trained layers and trains only the new output layers.
A pre-trained model has learned general features from a large dataset that transfer well to new tasks.
Mixed precision training uses both 16-bit and 32-bit floating point numbers to reduce memory and increase speed.
Gradient accumulation simulates larger batch sizes by accumulating gradients over multiple mini-batches before updating.
Distributed training splits the workload across multiple GPUs or machines.
Data parallelism replicates the model across devices and splits the data.
Model parallelism splits the model across devices when it is too large for a single GPU.

--------------------------------------------------------------------------------
2.3 State-Space Models (SSM)
--------------------------------------------------------------------------------

A state-space model represents a system using state variables and linear transformations.
The continuous-time SSM is defined by: dx/dt = Ax + Bu, y = Cx + Du.
A is the state transition matrix that determines how the hidden state evolves over time.
B is the input projection matrix that maps inputs into the hidden state.
C is the output projection matrix that maps the hidden state to the output.
D is the direct feedthrough matrix (often zero) that connects input directly to output.
Discretization converts the continuous SSM into a discrete form suitable for processing sequences.
The zero-order hold (ZOH) is a common discretization method for SSMs.
The bilinear transform is another discretization method that preserves frequency properties.

The Structured State Space for Sequences (S4) model was a breakthrough that made SSMs practical for deep learning.
S4 uses a specific parameterization of the A matrix based on the HiPPO framework.
HiPPO (High-order Polynomial Projection Operators) provides a theoretically grounded initialization for A.
The HiPPO initialization enables the model to efficiently remember and compress long histories.

SSMs achieve O(n) time complexity for sequence processing, compared to O(n^2) for self-attention in transformers.
During training, SSMs can process entire sequences in parallel using convolution.
The SSM convolution kernel is pre-computed from A, B, and C, enabling efficient parallel training.
During inference, SSMs process tokens one at a time using the recurrent form, giving constant time per step.
This dual mode (convolution for training, recurrence for inference) is a key advantage of SSMs.
SSMs maintain a fixed-size hidden state that summarizes all previous context.
The fixed-size state means memory usage does not grow with sequence length.
SSMs are particularly effective for tasks involving very long sequences like audio, DNA, and time series.

Mamba is an SSM variant that introduces input-dependent (selective) parameters.
In Mamba, the B, C, and delta parameters are functions of the input, allowing the model to filter information selectively.
Selective state spaces allow the model to decide what information to remember or forget based on context.
Mamba achieves performance competitive with transformers while maintaining linear scaling.
Mamba-2 further improves upon Mamba with structured state space duality (SSD).
The SSD framework shows a mathematical connection between SSMs and attention mechanisms.

SSMs handle long-range dependencies through their state transition dynamics rather than direct attention.
The eigenvalues of the A matrix determine how quickly information decays in the hidden state.
Diagonal SSMs constrain the A matrix to be diagonal, simplifying computation while maintaining expressiveness.
Multi-head SSMs process multiple channels independently, similar to multi-head attention.

--------------------------------------------------------------------------------
2.4 Knowledge Graphs
--------------------------------------------------------------------------------

A knowledge graph represents information as a network of entities and their relationships.
An entity is a real-world object, concept, or thing represented as a node in the graph.
A relation (or edge) describes the connection between two entities.
A triple (or triplet) is the fundamental unit of a knowledge graph: (subject, predicate, object).
For example: (Paris, is_capital_of, France) is a triple.
The subject is the entity the statement is about.
The predicate defines the relationship between the subject and object.
The object is the entity that the subject is related to.

RDF (Resource Description Framework) is a standard model for representing knowledge graph data.
SPARQL is a query language for retrieving and manipulating data in RDF knowledge graphs.
An ontology defines the types of entities, relationships, and constraints in a knowledge graph.
OWL (Web Ontology Language) is a semantic web language for defining rich ontologies.
A schema defines the allowed entity types and relation types in a knowledge graph.

Graph traversal is the process of visiting nodes in a graph by following edges.
Breadth-first search (BFS) explores all neighbors at the current depth before moving deeper.
Depth-first search (DFS) explores as far as possible along each branch before backtracking.
Shortest path algorithms find the minimum-cost path between two nodes.
Multi-hop reasoning answers questions that require following multiple edges in the knowledge graph.
For example, answering "Where was the president of France born?" requires two hops: country -> president -> birthplace.

Knowledge graph embedding represents entities and relations as vectors in a continuous space.
TransE models relations as translations in the embedding space: head + relation is approximately equal to tail.
DistMult uses a bilinear model with diagonal relation matrices.
ComplEx extends DistMult to complex-valued embeddings for asymmetric relations.
RotatE models relations as rotations in complex space.

Link prediction infers missing edges in a knowledge graph.
Entity resolution identifies different references to the same real-world entity.
Knowledge graph completion adds missing facts based on existing patterns.
Graph neural networks (GNNs) learn node representations by aggregating information from neighboring nodes.
R-GCN (Relational Graph Convolutional Network) extends GNNs to handle multiple relation types.

Named Entity Recognition (NER) identifies entities like persons, locations, and organizations in text.
Relation extraction identifies relationships between entities mentioned in text.
Knowledge graph construction pipelines combine NER, relation extraction, and entity resolution.
Wikidata is a free, collaborative knowledge base with millions of structured data items.
DBpedia extracts structured information from Wikipedia.
A property graph model allows both nodes and edges to have key-value properties.
Neo4j is the most popular graph database, using the property graph model and Cypher query language.

--------------------------------------------------------------------------------
2.5 Hebbian Learning
--------------------------------------------------------------------------------

Hebbian learning is based on the principle: neurons that fire together wire together.
Donald Hebb proposed this learning rule in 1949 in his book "The Organization of Behavior."
The basic Hebbian rule strengthens the connection between two neurons when both are simultaneously active.
Mathematically, the weight update is: delta_w = eta * x_i * x_j, where eta is the learning rate.
Hebbian learning is unsupervised; it does not require labeled data or error signals.
Co-occurrence strengthening means that patterns frequently seen together form stronger associations.

Anti-Hebbian learning weakens connections between neurons that are not co-active.
Oja's rule is a normalized version of Hebbian learning that prevents weights from growing unboundedly.
Competitive learning allows only the most active neuron to update its weights, forming winner-take-all circuits.
Self-organizing maps (SOMs) use competitive Hebbian learning to create topological maps of input data.

Spike-Timing-Dependent Plasticity (STDP) extends Hebbian learning by considering the precise timing of neural spikes.
If the pre-synaptic neuron fires before the post-synaptic neuron, the connection is strengthened (long-term potentiation).
If the pre-synaptic neuron fires after the post-synaptic neuron, the connection is weakened (long-term depression).

In the context of AI, Hebbian learning can strengthen knowledge graph edges based on concept co-occurrence.
When two concepts frequently appear together in queries or text, their connection weight increases.
This enables the system to learn associative patterns without explicit supervision.
Hebbian-style updates can be used to prioritize frequently accessed knowledge paths.
Decay mechanisms reduce connection strength over time for rarely co-occurring concepts.
Hebbian learning complements gradient-based learning by capturing statistical regularities in usage patterns.

--------------------------------------------------------------------------------
2.6 Transformers vs SSM Comparison
--------------------------------------------------------------------------------

The Transformer architecture, introduced in 2017, uses self-attention to process sequences.
Self-attention computes a weighted sum of all positions, allowing each token to attend to every other token.
The attention mechanism has O(n^2) time and memory complexity with respect to sequence length n.
Transformers excel at capturing long-range dependencies through direct attention connections.
Multi-head attention runs multiple attention operations in parallel and concatenates the results.
Positional encoding adds position information since attention itself is permutation-invariant.

SSMs process sequences with O(n) time complexity, making them more efficient for long sequences.
SSMs maintain a fixed-size hidden state, giving O(1) memory per generation step during inference.
Transformers require storing all previous key-value pairs during autoregressive generation, using O(n) memory.
The KV-cache in transformers grows linearly with sequence length and becomes a memory bottleneck.

Transformers are currently dominant in language modeling due to their flexibility and training efficiency.
SSMs have shown strong results on long-range benchmarks like the Long Range Arena.
Hybrid architectures combine transformer layers with SSM layers for the best of both approaches.
Some hybrid models use SSM layers for long-range context and attention layers for precise local patterns.

Transformers can be parallelized efficiently during training because attention is computed over the full sequence.
SSMs can also be parallelized during training via the convolution view.
During inference, SSMs have a clear advantage: constant time and memory per token versus growing requirements for transformers.
For tasks with very long contexts (books, audio, genomics), SSMs are increasingly preferred.
For tasks requiring precise retrieval of specific details from context, attention mechanisms remain superior.

--------------------------------------------------------------------------------
2.7 Model Deployment
--------------------------------------------------------------------------------

Model quantization reduces the numerical precision of model weights to decrease memory usage and increase speed.
INT8 quantization converts 32-bit floating-point weights to 8-bit integers, reducing model size by approximately 4x.
INT4 quantization uses 4-bit integers for even greater compression, with some accuracy loss.
Post-training quantization applies quantization to an already trained model without retraining.
Quantization-aware training simulates quantization during training, resulting in better accuracy after quantization.
GPTQ is a popular post-training quantization method for large language models.
AWQ (Activation-aware Weight Quantization) preserves important weights based on activation patterns.

ONNX (Open Neural Network Exchange) is an open format for representing machine learning models.
ONNX enables model portability across different frameworks (PyTorch, TensorFlow, etc.).
ONNX Runtime is a high-performance inference engine that optimizes ONNX models for deployment.
TensorRT is NVIDIA's inference optimizer and runtime for GPU-accelerated deployment.
CoreML is Apple's framework for deploying ML models on iOS, macOS, and other Apple platforms.

Model serving is the process of hosting a trained model to handle prediction requests.
TorchServe is a PyTorch model serving framework with features like multi-model serving and logging.
TensorFlow Serving is a flexible, high-performance system for serving TensorFlow models.
Triton Inference Server by NVIDIA supports multiple frameworks and provides dynamic batching.
vLLM is an efficient inference engine for large language models using PagedAttention.
Dynamic batching groups incoming requests into batches to maximize GPU utilization.
Model warm-up pre-loads the model and runs dummy predictions to eliminate cold start latency.

KV-cache optimization reduces the memory footprint of autoregressive transformer inference.
PagedAttention manages KV-cache memory in pages, similar to operating system virtual memory.
Speculative decoding uses a small draft model to predict multiple tokens, verified by the larger model.
Continuous batching processes new requests as they arrive rather than waiting for a full batch.
Tensor parallelism splits model layers across multiple GPUs for faster inference.
Pipeline parallelism splits model layers sequentially across GPUs, with each GPU processing a stage.

Knowledge distillation trains a smaller student model to mimic the behavior of a larger teacher model.
The student model learns from the soft probability distributions produced by the teacher.
Pruning removes unnecessary weights or neurons from a model to reduce its size.
Structured pruning removes entire channels or layers, maintaining hardware-friendly structure.
Unstructured pruning removes individual weights, resulting in sparse matrices.

--------------------------------------------------------------------------------
2.8 NOVA Architecture
--------------------------------------------------------------------------------

NOVA is an AI architecture that combines four key components: SSM, Knowledge Graph, Template Engine, and Hebbian Learning.
The SSM (State-Space Model) component provides efficient sequential reasoning with O(n) complexity.
The Knowledge Graph stores structured world knowledge as entities and relationships.
The Template Engine generates responses using structured templates informed by retrieved knowledge.
Hebbian Learning dynamically strengthens connections between frequently co-occurring concepts.

In NOVA, user input is first processed by the SSM layer to understand context and intent.
The SSM output activates relevant regions of the knowledge graph for targeted retrieval.
Graph traversal follows relationship edges to gather related facts and context.
Hebbian weights on graph edges prioritize paths that have been frequently useful.
The template engine uses retrieved knowledge and SSM context to generate structured responses.

NOVA's SSM component maintains a compressed representation of conversation history.
Unlike transformers, NOVA does not need to store growing key-value caches.
The knowledge graph provides explicit, interpretable reasoning paths.
When the system answers a question, the reasoning path through the graph can be inspected.
This makes NOVA more interpretable than pure neural approaches.

Hebbian learning in NOVA updates graph edge weights based on usage patterns.
If a query about Python frequently triggers retrieval of documentation nodes, those connections strengthen.
Rarely accessed connections gradually weaken through decay, keeping the graph efficient.
This adaptive weighting allows NOVA to personalize its knowledge retrieval over time.

The template engine ensures consistent, well-structured output.
Templates can be domain-specific, with different formats for code, explanations, and conversations.
Template selection is guided by the SSM's understanding of the user's intent.
Slot filling populates template variables with retrieved knowledge and generated content.

NOVA's architecture is designed to be efficient enough to run on edge devices.
The SSM requires fixed-size state (typically a few kilobytes), not scaling with context length.
The knowledge graph can be partitioned and loaded on demand.
Quantized models further reduce memory requirements for deployment.
The combination of these components allows NOVA to provide knowledgeable, efficient, and interpretable AI.

3.1 How to Build a REST API
--------------------------------------------------------------------------------

Step 1: Define your resource model and API endpoints before writing any code.
List all resources (e.g., users, products, orders) and the operations needed on each.
Map operations to HTTP methods: GET for read, POST for create, PUT/PATCH for update, DELETE for remove.
Design URL patterns following REST conventions: /resources for collections, /resources/{id} for individual items.

Step 2: Choose a framework appropriate to your language.
Express.js is the most popular Node.js framework for building REST APIs.
Flask and FastAPI are popular Python frameworks; FastAPI provides automatic documentation.
Spring Boot is the standard Java framework for building REST APIs.
Gin and Echo are popular Go frameworks for high-performance APIs.

Step 3: Set up the project structure with clear separation of concerns.
Organize code into layers: routes (endpoints), controllers (request handling), services (business logic), and models (data).
Use a configuration file or environment variables for database URLs, API keys, and other settings.
Install necessary dependencies: the framework, database driver, validation library, and testing tools.

Step 4: Define your data models and set up the database connection.
Create model definitions that map to your database tables or collections.
Use an ORM (Object-Relational Mapping) like Sequelize, SQLAlchemy, or Hibernate for relational databases.
Define validation rules on your models to ensure data integrity.

Step 5: Implement CRUD operations for each resource.
The Create operation accepts a POST request, validates the input, and inserts a new record.
The Read operation accepts a GET request and returns one or more resources.
The Update operation accepts a PUT or PATCH request, validates the input, and modifies the existing record.
The Delete operation accepts a DELETE request and removes the record.
Always return appropriate HTTP status codes: 201 for creation, 200 for success, 404 for not found.

Step 6: Add input validation and error handling.
Validate all incoming data before processing it; never trust client input.
Return descriptive error messages with appropriate status codes (400 for bad input, 422 for validation failure).
Use a global error handler to catch unhandled exceptions and return 500 with a generic message.
Never expose stack traces or internal details in production error responses.

Step 7: Add authentication and authorization middleware.
Implement middleware that verifies tokens or session cookies on protected routes.
Return 401 Unauthorized if no valid credentials are provided.
Return 403 Forbidden if the user is authenticated but lacks permission for the action.

Step 8: Add pagination, filtering, and sorting for list endpoints.
Accept query parameters like page, limit, sort, and filter fields.
Return pagination metadata including total count, current page, and total pages.
Set reasonable default and maximum values for page size to prevent abuse.

Step 9: Write tests for every endpoint.
Test successful operations, validation failures, authentication errors, and edge cases.
Use an in-memory database or test containers for isolated integration tests.
Automate tests in your CI/CD pipeline to run on every code change.

Step 10: Document your API using OpenAPI/Swagger.
Define request and response schemas, parameters, and error codes.
Provide example requests and responses for each endpoint.
Host interactive documentation so consumers can try endpoints directly.

--------------------------------------------------------------------------------
3.2 How to Implement Authentication
--------------------------------------------------------------------------------

Authentication verifies the identity of a user; authorization determines what they can access.
The most common approaches are session-based auth, token-based auth (JWT), and OAuth 2.0.

For session-based authentication:
The user submits credentials (username and password) to a login endpoint.
The server verifies the credentials against stored hashed passwords.
On success, the server creates a session and stores the session ID in a server-side store.
The session ID is sent to the client in a cookie with HttpOnly, Secure, and SameSite flags.
On subsequent requests, the browser automatically sends the cookie; the server looks up the session.
Session expiration should be configured to limit how long a session remains valid.
Implement session renewal on activity and absolute timeout after a maximum duration.

For JWT-based authentication:
The user submits credentials to a login endpoint.
The server verifies credentials and generates a JWT containing the user ID and roles.
The JWT is signed with a secret key (HS256) or private key (RS256).
The client stores the JWT (in memory or an HttpOnly cookie, not localStorage).
On each request, the client sends the JWT in the Authorization header as "Bearer <token>".
The server validates the JWT signature and checks the expiration claim.
Access tokens should be short-lived (15 minutes); refresh tokens can be longer-lived (days or weeks).
Implement a token refresh endpoint that accepts a refresh token and returns a new access token.
Store refresh tokens securely and implement rotation to detect token theft.

Password storage best practices:
Never store passwords in plain text.
Use bcrypt, scrypt, or Argon2 to hash passwords with a unique salt per user.
Enforce minimum password length (at least 8 characters) and check against known breached passwords.
Implement rate limiting on login endpoints to prevent brute-force attacks.
Lock accounts temporarily after a configurable number of failed login attempts.

Multi-factor authentication (MFA) adds an additional verification step.
TOTP (Time-based One-Time Password) generates a 6-digit code that changes every 30 seconds.
The server stores a shared secret; the user's authenticator app generates codes from the same secret.
SMS-based MFA sends a code via text message but is less secure due to SIM swapping risks.
WebAuthn enables passwordless authentication using hardware security keys or biometrics.

Implement logout by invalidating the session or adding the JWT to a deny list.
Use HTTPS for all authentication endpoints to protect credentials in transit.
Set appropriate CORS headers to prevent cross-origin credential theft.

--------------------------------------------------------------------------------
3.3 How to Handle Database Migrations
--------------------------------------------------------------------------------

Database migrations are versioned scripts that modify the database schema over time.
Migrations allow the database schema to evolve alongside application code in a controlled manner.
Each migration has an "up" function (apply the change) and a "down" function (revert the change).
Migrations are numbered or timestamped to ensure they run in the correct order.

Popular migration tools include Flyway (Java), Alembic (Python), Knex (Node.js), and Goose (Go).
Rails ActiveRecord Migrations is the gold standard that inspired many migration tools.
Liquibase supports XML, YAML, JSON, and SQL formats for defining database changes.

Migration best practices:
Each migration should make a single, focused schema change.
Always write both up and down migrations so changes can be rolled back.
Never modify a migration that has already been applied to production.
Test migrations against a copy of production data before applying them to production.
Store migration files in version control alongside the application code.
Run migrations automatically as part of the deployment pipeline.
Use transactions in migrations so that a failed migration does not leave the schema in a partial state.
Add columns as nullable or with defaults before making them required, to avoid downtime.
Create indexes concurrently when possible to avoid locking the table during index creation.
For large tables, consider batched data migrations to avoid long-running transactions.

A migration table tracks which migrations have been applied to the database.
This table typically stores the migration version number, name, and the timestamp when it was applied.
Zero-downtime migrations require careful sequencing: add new columns first, deploy code, then remove old columns.
Backward-compatible migrations ensure that both old and new code versions can work with the schema.

--------------------------------------------------------------------------------
3.4 How to Set Up Logging and Monitoring
--------------------------------------------------------------------------------

Logging records events and data during application execution for debugging and auditing.
Use structured logging (JSON format) rather than plain text for machine-parseable logs.
Include a timestamp, log level, message, request ID, and relevant context in every log entry.
Standard log levels from least to most severe: TRACE, DEBUG, INFO, WARN, ERROR, FATAL.
Use DEBUG for detailed development information, INFO for normal operations, WARN for potential issues.
Use ERROR for failures that need attention, and FATAL for unrecoverable errors.
Never log sensitive data like passwords, tokens, credit card numbers, or personal information.

Centralize logs from all services into a single system for search and analysis.
The ELK stack (Elasticsearch, Logstash, Kibana) is a popular open-source logging solution.
Fluentd and Fluent Bit are lightweight log collectors that can forward logs to various destinations.
Loki is a log aggregation system designed to be cost-effective, paired with Grafana for visualization.
Cloud-native options include AWS CloudWatch, GCP Cloud Logging, and Azure Monitor.

For application monitoring:
Expose application metrics using a library like Prometheus client or StatsD.
Track the four golden signals: latency, traffic, errors, and saturation.
Measure request duration histograms to understand response time distribution.
Count HTTP requests by status code and endpoint to detect error rate spikes.
Monitor resource usage: CPU, memory, disk I/O, and network.
Set up alerts for anomalies: error rate exceeding 1%, latency exceeding the P99 threshold, or disk usage above 80%.
Use dashboards in Grafana or similar tools to visualize metrics in real-time.
Implement health check endpoints that monitoring systems can poll to verify service availability.

For distributed tracing:
Propagate trace IDs across all service calls using headers like traceparent or X-Request-ID.
Use OpenTelemetry, Jaeger, or Zipkin to collect and visualize distributed traces.
Traces show the full path of a request across services, including timing for each span.
Spans represent individual units of work within a trace, such as a database query or an API call.
Correlate logs, metrics, and traces using shared identifiers for comprehensive observability.

--------------------------------------------------------------------------------
3.5 How to Implement Caching
--------------------------------------------------------------------------------

Caching stores frequently accessed data in a fast-access layer to reduce latency and database load.
The cache-aside (lazy loading) pattern loads data into the cache only when it is first requested.
In cache-aside: check the cache first; on miss, load from the database, store in cache, then return.
Write-through caching writes data to both the cache and the database simultaneously.
Write-behind (write-back) caching writes to the cache immediately and asynchronously updates the database.
Read-through caching has the cache itself load data from the database on a miss.

Redis is an in-memory data store commonly used for caching, supporting strings, lists, sets, and hashes.
Memcached is a simpler distributed memory caching system focused on key-value storage.
Application-level caching stores computed results in memory (e.g., function memoization).
HTTP caching uses Cache-Control, ETag, and Last-Modified headers to cache responses at the client or CDN.

Cache invalidation strategies:
Time-based expiration (TTL) automatically removes cache entries after a set duration.
Event-based invalidation removes cache entries when the underlying data changes.
Cache-aside with TTL is the simplest approach but may serve stale data within the TTL window.
Publish-subscribe patterns can broadcast invalidation events to all cache instances.

Cache eviction policies determine what to remove when the cache is full.
LRU (Least Recently Used) evicts the entry that was accessed least recently.
LFU (Least Frequently Used) evicts the entry with the lowest access count.
FIFO (First In First Out) evicts the oldest entry.

Common caching pitfalls:
Cache stampede occurs when many requests simultaneously try to rebuild an expired cache entry.
Use locking or "stale while revalidate" patterns to prevent cache stampedes.
Cold cache after a restart can cause a sudden spike in database load; warm the cache during deployment.
Over-caching wastes memory; only cache data that is read frequently and changes infrequently.
Always set a TTL to prevent stale data from persisting indefinitely.

--------------------------------------------------------------------------------
3.6 How to Handle File Uploads
--------------------------------------------------------------------------------

Accept file uploads via multipart/form-data encoding in HTTP POST requests.
Set a maximum file size limit on the server to prevent abuse (e.g., 10 MB for images, 100 MB for videos).
Validate the file type by checking both the Content-Type header and the file's magic bytes (not just the extension).
Generate a unique filename (using UUID or hash) to prevent collisions and path traversal attacks.
Never store uploaded files using the original filename provided by the user.

For small applications, store files on the local filesystem in a dedicated uploads directory.
For production, use object storage like AWS S3, Google Cloud Storage, or Azure Blob Storage.
Object storage provides durability, scalability, CDN integration, and access control.
Use pre-signed URLs to allow clients to upload directly to object storage, reducing server load.
Pre-signed URLs are time-limited and grant temporary permission to upload to a specific path.

Process uploaded files asynchronously when possible (e.g., image resizing, virus scanning).
Use a message queue to decouple file processing from the upload request.
Store file metadata (name, size, type, storage path, upload timestamp) in the database.
Return the file's metadata and access URL in the upload response.
Implement virus scanning for user-uploaded files before making them accessible.
Set appropriate Content-Disposition headers when serving files to control download behavior.
Use a CDN to serve frequently accessed files for better performance and reduced server load.
Implement cleanup routines to remove orphaned files that are no longer referenced in the database.

--------------------------------------------------------------------------------
3.7 How to Implement Pagination
--------------------------------------------------------------------------------

Pagination divides large result sets into smaller, manageable pages.
Offset-based pagination uses page number and page size: SELECT * FROM items LIMIT 20 OFFSET 40.
Offset pagination is simple but becomes slow for large offsets because the database must scan all skipped rows.
Cursor-based (keyset) pagination uses a pointer to the last item returned: WHERE id > last_seen_id LIMIT 20.
Cursor pagination is more efficient for large datasets because it uses an indexed column.
Cursor pagination does not support jumping to arbitrary pages; it only supports next and previous.

Return pagination metadata in the response:
Include total_count (total number of items matching the query).
Include page and per_page for offset-based pagination.
Include next_cursor and prev_cursor for cursor-based pagination.
Include has_next and has_previous boolean flags.
Include links to the next and previous pages (HATEOAS-style).

Set a default page size (e.g., 20) and a maximum page size (e.g., 100) to prevent abuse.
Validate that page and per_page parameters are positive integers within allowed bounds.
For cursor-based pagination, encode the cursor as an opaque string (e.g., base64-encoded) so clients do not depend on its format.
When sorting by a non-unique column, include a tiebreaker (like ID) to ensure stable ordering.
Combine pagination with filtering and sorting, applying filters before pagination in the query.

--------------------------------------------------------------------------------
3.8 How to Implement Search
--------------------------------------------------------------------------------

Full-text search allows users to find records by matching words and phrases in text fields.
Elasticsearch is the most popular open-source search engine, built on Apache Lucene.
Elasticsearch stores data in indexes and supports complex queries, aggregations, and fuzzy matching.
PostgreSQL provides built-in full-text search using tsvector and tsquery types.
Typesense and Meilisearch are lightweight, easy-to-deploy search engines.

Indexing for search:
Tokenization splits text into individual words (tokens).
Stemming reduces words to their root form (e.g., "running" becomes "run").
Stop word removal filters out common words like "the," "is," and "at."
An inverted index maps each token to the list of documents containing it.

Search query features:
Boolean queries combine terms with AND, OR, and NOT operators.
Phrase queries match exact sequences of words.
Fuzzy matching finds results even when the search term has typos or misspellings.
Wildcard queries match patterns with * and ? placeholders.
Faceted search provides counts of results grouped by categories (e.g., brand, price range).
Autocomplete suggests search terms as the user types, using prefix matching or n-gram analysis.
Relevance scoring ranks results by how well they match the query, typically using BM25 or TF-IDF.

Search implementation steps:
Sync data from your primary database to the search engine (using change data capture or scheduled jobs).
Build a search API endpoint that accepts query, filters, sort, and pagination parameters.
Return highlighted snippets showing where the query matched in the results.
Implement search analytics to track popular queries and zero-result queries.
Use synonyms to expand queries (e.g., "laptop" also matches "notebook computer").

--------------------------------------------------------------------------------
3.9 How to Implement WebSockets
--------------------------------------------------------------------------------

WebSocket is a protocol that provides full-duplex communication over a single TCP connection.
Unlike HTTP, WebSocket allows the server to push data to the client without the client requesting it.
A WebSocket connection starts with an HTTP upgrade request from the client.
The server responds with HTTP 101 Switching Protocols to establish the WebSocket connection.
Once established, the connection remains open and both sides can send messages freely.

Common WebSocket use cases include chat applications, live notifications, real-time dashboards, and collaborative editing.
Socket.IO is a popular library that provides WebSocket functionality with automatic fallbacks and reconnection.
The ws library is a lightweight WebSocket implementation for Node.js.
Django Channels adds WebSocket support to Django applications.
Spring WebSocket provides WebSocket support for Java applications.

WebSocket implementation considerations:
Implement heartbeat (ping/pong) messages to detect and clean up dead connections.
Handle connection drops gracefully with automatic reconnection logic on the client.
Authenticate WebSocket connections during the initial HTTP upgrade handshake.
Use rooms or channels to group related connections (e.g., all users in a chat room).
Implement message acknowledgments for reliable delivery of critical messages.
Scale WebSocket servers horizontally using a pub/sub system like Redis to broadcast messages across instances.
Consider Server-Sent Events (SSE) as a simpler alternative when only server-to-client streaming is needed.
SSE uses standard HTTP and is easier to implement but only supports one-way communication.

--------------------------------------------------------------------------------
3.10 How to Deploy to Production
--------------------------------------------------------------------------------

Pre-deployment checklist:
Ensure all tests pass and code has been reviewed.
Remove debug logging, console.log statements, and development-only code.
Set environment variables for production configuration (database URL, API keys, log levels).
Enable HTTPS for all endpoints.
Configure proper CORS headers for your production domains.
Set up error tracking with a service like Sentry to capture and alert on production errors.

Container-based deployment:
Build a production Docker image with multi-stage builds to minimize image size.
Tag images with the Git commit hash or semantic version for traceability.
Push the image to a container registry (Docker Hub, ECR, GCR, ACR).
Deploy to Kubernetes using Deployments with rolling update strategy.
Configure resource requests and limits (CPU and memory) for each container.
Set up liveness and readiness probes so Kubernetes knows when a container is healthy.

Database considerations:
Run database migrations before deploying new application code.
Back up the database before applying migrations.
Use connection pooling to manage database connections efficiently.
Ensure the application handles database connection failures gracefully with retries.

Post-deployment verification:
Run smoke tests against the production environment to verify critical paths.
Monitor error rates, latency, and throughput immediately after deployment.
Have a rollback plan ready: keep the previous version's image available for quick rollback.
Use canary deployments to gradually shift traffic and catch issues before full rollout.
Document the deployment process so any team member can perform it.
Automate as much of the deployment process as possible to reduce human error.

SECTION 4: BUSINESS AND ENTERPRISE

--------------------------------------------------------------------------------
4.1 SaaS Architecture Patterns
--------------------------------------------------------------------------------

SaaS (Software as a Service) delivers applications over the internet on a subscription basis.
The key SaaS architectural concerns are multi-tenancy, scalability, billing, and security.
A SaaS application must isolate tenant data while sharing infrastructure to keep costs manageable.
SaaS onboarding should be self-service, allowing users to sign up and start using the product immediately.
A free trial or freemium tier reduces the barrier to adoption and drives growth.

The application layer should be stateless to enable horizontal scaling.
Session state should be stored externally in Redis or a similar shared store.
Static assets should be served through a CDN for global low-latency access.
Background jobs should run on separate worker processes or serverless functions.
Use feature flags to enable different features for different subscription tiers.

SaaS products typically offer multiple pricing tiers with different feature sets and usage limits.
Usage-based pricing charges customers based on actual consumption (API calls, storage, compute time).
Per-seat pricing charges based on the number of users on the account.
Flat-rate pricing charges a fixed monthly or annual fee regardless of usage.
Annual contracts with discounts encourage longer commitments and improve revenue predictability.
Revenue metrics to track include Monthly Recurring Revenue (MRR), Annual Recurring Revenue (ARR), and churn rate.
Customer Acquisition Cost (CAC) is the total cost to acquire a new customer.
Lifetime Value (LTV) is the total revenue expected from a customer over their entire relationship.
A healthy SaaS business targets an LTV-to-CAC ratio of at least 3:1.

SaaS applications must handle graceful degradation when downstream services fail.
Circuit breakers prevent cascading failures across microservices.
Rate limiting protects shared infrastructure from noisy tenants.
Tenant-aware monitoring tracks performance and usage metrics per tenant.

--------------------------------------------------------------------------------
4.2 Multi-Tenant Design
--------------------------------------------------------------------------------

Multi-tenancy allows a single application instance to serve multiple customers (tenants).
Each tenant's data must be isolated to prevent unauthorized access across tenants.

Database-level isolation strategies:
Separate databases per tenant provide the strongest isolation but highest operational overhead.
Separate schemas per tenant within a shared database balance isolation with manageable overhead.
Shared tables with a tenant_id column are the most cost-effective but require careful access control.
Row-level security (RLS) in PostgreSQL enforces tenant isolation at the database level automatically.

Application-level isolation:
Every database query must include the tenant context to prevent data leakage.
Middleware should extract the tenant identifier from the request (subdomain, header, or JWT claim).
Use a tenant context object that is available throughout the request lifecycle.
Automated tests should verify that one tenant cannot access another tenant's data.

Performance isolation:
A "noisy neighbor" problem occurs when one tenant's heavy usage impacts others.
Per-tenant rate limiting prevents any single tenant from consuming excessive resources.
Connection pooling should be configured to prevent one tenant from exhausting all database connections.
Dedicated infrastructure can be offered to enterprise tenants who need guaranteed performance.

Tenant provisioning:
Automate the creation of new tenant resources (database schemas, storage buckets, configuration).
Use database migrations that work correctly in a multi-tenant environment.
Support tenant-specific configuration for branding, feature flags, and integrations.
Implement tenant deprovisioning to clean up resources when a tenant leaves.

--------------------------------------------------------------------------------
4.3 Rate Limiting Strategies
--------------------------------------------------------------------------------

Rate limiting controls the number of requests a client can make within a time window.
Rate limiting protects APIs from abuse, prevents denial-of-service attacks, and ensures fair resource sharing.

Common rate limiting algorithms:
Fixed window counts requests in fixed time intervals (e.g., 100 requests per minute).
Fixed window can allow burst traffic at the boundary between two windows.
Sliding window log tracks the timestamp of each request and counts requests within the sliding window.
Sliding window counter combines fixed window counts with interpolation for more accurate limiting.
Token bucket starts with a fixed number of tokens; each request consumes a token, and tokens refill at a steady rate.
Token bucket allows burst traffic up to the bucket capacity while enforcing an average rate.
Leaky bucket processes requests at a fixed rate, queuing excess requests.

Rate limit identification:
Identify clients by API key, user ID, IP address, or a combination.
Apply different rate limits to different API endpoints based on their cost and sensitivity.
Offer higher rate limits to paid tiers and lower limits to free tiers.

Rate limit headers to include in responses:
X-RateLimit-Limit: the maximum number of requests allowed in the window.
X-RateLimit-Remaining: the number of requests remaining in the current window.
X-RateLimit-Reset: the time when the rate limit window resets.
Return HTTP 429 Too Many Requests when the limit is exceeded.
Include a Retry-After header telling the client how long to wait.

Distributed rate limiting:
Use Redis or a similar shared store for rate limit counters across multiple server instances.
Redis INCR and EXPIRE commands provide atomic counter operations with automatic expiration.
Consider eventual consistency in distributed rate limiting; slight over-allowance is usually acceptable.

--------------------------------------------------------------------------------
4.4 Billing and Subscription Models
--------------------------------------------------------------------------------

A billing system manages subscription lifecycle, invoicing, payments, and revenue recognition.
Stripe, Braintree, and Paddle are popular payment processing platforms for SaaS.
Never store raw credit card numbers; use the payment processor's tokenization service.
PCI DSS (Payment Card Industry Data Security Standard) defines requirements for handling card data.

Subscription lifecycle events:
Trial started: the customer begins a free trial period.
Subscription created: the customer selects a plan and provides payment information.
Payment succeeded: the recurring charge was processed successfully.
Payment failed: the charge was declined; retry logic should attempt again with exponential backoff.
Subscription upgraded or downgraded: the customer changes their plan.
Proration calculates partial charges or credits when a plan changes mid-billing-cycle.
Subscription cancelled: the customer cancels; provide access until the end of the billing period.
Subscription expired: access is revoked after the cancellation period ends.

Use webhook events from the payment processor to keep your system synchronized.
Process webhooks idempotently because payment processors may send the same event multiple times.
Store a complete audit trail of all billing events for compliance and dispute resolution.
Provide customers with self-service billing management: update payment method, view invoices, change plans.
Send invoice emails and payment failure notifications promptly.
Implement dunning management: a series of escalating notifications for failed payments before suspension.
Support annual billing with discounts as an incentive for longer commitments.
Handle currency conversion and localized pricing for international customers.

--------------------------------------------------------------------------------
4.5 Audit Logging Requirements
--------------------------------------------------------------------------------

An audit log records who did what, when, and on which resource within the system.
Audit logs are essential for security investigations, compliance, and debugging.
Every audit log entry should include: timestamp, actor (user ID), action, resource type, resource ID, and outcome.
Include the IP address and user agent of the request for security analysis.
Record both successful and failed actions (especially failed login attempts and permission denials).

Audit logs must be immutable; once written, they should not be modified or deleted.
Write audit logs to an append-only store, separate from the application database.
Use a dedicated logging service or event stream (e.g., Kafka) for audit events.
Retain audit logs for the period required by your compliance framework (often 1-7 years).
Implement log rotation and archival to manage storage costs for long retention periods.

Critical events to audit:
Authentication events: login, logout, failed login attempts, password changes, MFA enrollment.
Authorization events: permission grants, role changes, access denials.
Data access: reads of sensitive records (especially PII and financial data).
Data modification: creates, updates, and deletes of business-critical records.
Configuration changes: system settings, feature flags, API key creation or revocation.
Administrative actions: user creation, account suspension, billing changes.

Provide an audit log viewer in the admin interface with filtering by user, action, resource, and time range.
Make audit logs searchable and exportable for compliance auditors.
Alert on suspicious patterns like multiple failed logins, unusual data access volumes, or off-hours admin actions.

--------------------------------------------------------------------------------
4.6 GDPR and Compliance Basics
--------------------------------------------------------------------------------

GDPR (General Data Protection Regulation) is a European Union regulation protecting personal data and privacy.
GDPR applies to any organization that processes personal data of EU residents, regardless of the organization's location.
Personal data is any information that can identify a natural person, directly or indirectly.
The data controller determines the purposes and means of processing personal data.
The data processor processes personal data on behalf of the controller.

Key GDPR principles:
Lawfulness: data processing must have a legal basis (consent, contract, legitimate interest, etc.).
Purpose limitation: data must be collected for specified, explicit, and legitimate purposes.
Data minimization: only collect data that is necessary for the stated purpose.
Accuracy: personal data must be accurate and kept up to date.
Storage limitation: data must not be kept longer than necessary.
Integrity and confidentiality: data must be protected against unauthorized access and accidental loss.

Data subject rights under GDPR:
Right of access: individuals can request a copy of their personal data.
Right to rectification: individuals can request correction of inaccurate data.
Right to erasure (right to be forgotten): individuals can request deletion of their data.
Right to data portability: individuals can request their data in a machine-readable format.
Right to object: individuals can object to processing based on legitimate interest or direct marketing.
Right to restrict processing: individuals can request that processing be limited.

Data breach notification: breaches must be reported to the supervisory authority within 72 hours.
Data Protection Impact Assessment (DPIA) is required for high-risk processing activities.
A Data Protection Officer (DPO) may be required for organizations conducting large-scale data processing.
Privacy by design means incorporating data protection into the development process from the start.
Implement data retention policies that automatically delete or anonymize data after the retention period.
Maintain a Record of Processing Activities (ROPA) documenting all data processing operations.
Consent must be freely given, specific, informed, and unambiguous; pre-ticked checkboxes are not valid consent.

SOC 2 (Service Organization Control 2) is an auditing standard for service providers managing customer data.
SOC 2 evaluates controls related to security, availability, processing integrity, confidentiality, and privacy.
HIPAA (Health Insurance Portability and Accountability Act) governs the handling of health information in the United States.
PCI DSS governs the handling of payment card data.
ISO 27001 is an international standard for information security management systems.

--------------------------------------------------------------------------------
4.7 High Availability Patterns
--------------------------------------------------------------------------------

High availability (HA) ensures that a system remains operational and accessible despite component failures.
Availability is typically measured as a percentage of uptime: 99.9% (three nines) allows 8.76 hours of downtime per year.
99.99% availability (four nines) allows only 52.6 minutes of downtime per year.
99.999% availability (five nines) allows only 5.26 minutes of downtime per year.

Redundancy eliminates single points of failure by duplicating critical components.
Active-active redundancy runs multiple instances simultaneously, all handling traffic.
Active-passive redundancy keeps a standby instance that takes over when the active instance fails.
A health check monitors the status of each instance and routes traffic away from unhealthy ones.

Database high availability:
Primary-replica replication provides read scalability and failover capability.
Synchronous replication ensures replicas are always up to date but adds latency.
Asynchronous replication has lower latency but may lose recently committed data during failover.
Automatic failover promotes a replica to primary when the primary becomes unavailable.
Multi-region database replication provides resilience against entire region outages.

Load balancer high availability:
Deploy load balancers in pairs with automatic failover.
Use DNS-based failover to route traffic to a backup load balancer if the primary fails.
Cloud load balancers (ALB, NLB) are managed and inherently highly available.

Application-level patterns:
Retry with exponential backoff handles transient failures gracefully.
Circuit breakers prevent cascading failures by stopping requests to failing services.
Bulkhead isolation limits the impact of a failure to one part of the system.
Graceful degradation serves reduced functionality rather than complete failure.
Queue-based load leveling uses message queues to absorb traffic spikes.

Health checks and monitoring:
Implement both liveness checks (is the process running?) and readiness checks (can it handle requests?).
Use synthetic monitoring (periodic test requests) to detect issues before users report them.
Configure alerts with appropriate thresholds and escalation policies.
Conduct regular game days or chaos engineering exercises to test failover procedures.

--------------------------------------------------------------------------------
4.8 Disaster Recovery
--------------------------------------------------------------------------------

Disaster recovery (DR) is the process of restoring systems and data after a catastrophic failure.
Recovery Time Objective (RTO) is the maximum acceptable time to restore service after a disaster.
Recovery Point Objective (RPO) is the maximum acceptable amount of data loss, measured in time.
A lower RTO and RPO require more investment in infrastructure and automation.

Backup strategies:
Full backups copy all data; they are comprehensive but time-consuming and storage-intensive.
Incremental backups copy only data that changed since the last backup.
Differential backups copy all data changed since the last full backup.
Store backups in a different geographic region from the primary data.
Encrypt backups at rest and in transit.
Test backup restoration regularly; an untested backup is not a backup.
Automate backup creation and verification as part of your infrastructure.

DR strategies by RTO:
Backup and restore (hours): cheapest but slowest; restore from backups when disaster occurs.
Pilot light (minutes to hours): keep core services minimally running in the DR region.
Warm standby (minutes): run a scaled-down copy of the full environment in the DR region.
Multi-site active-active (near zero): run full copies in multiple regions, all handling production traffic.

DR planning:
Document the DR plan with step-by-step runbooks for each failure scenario.
Assign clear roles and responsibilities for DR execution.
Conduct DR drills at least annually to validate the plan and train the team.
Keep DR documentation up to date as the architecture evolves.
Include communication plans for notifying stakeholders during an outage.
Maintain an inventory of all critical systems, their dependencies, and their priority for recovery.
Use infrastructure as code to ensure the DR environment can be rapidly provisioned.
DNS failover can automatically redirect traffic to the DR region during an outage.
Data replication lag must be monitored to ensure RPO can be met during failover.

--------------------------------------------------------------------------------
5.1 Common Kubernetes Issues
--------------------------------------------------------------------------------

OOMKilled means a container exceeded its memory limit and was terminated by the kernel.
To diagnose OOMKilled: check the container's memory limit, analyze memory usage patterns, and look for memory leaks.
Increase the memory limit if the application legitimately needs more memory.
Use memory profiling tools to identify and fix memory leaks in the application code.
Java applications may need explicit JVM heap size configuration (-Xmx) within the container memory limit.

CrashLoopBackOff means a container keeps crashing and Kubernetes is restarting it with increasing backoff delays.
To diagnose CrashLoopBackOff: check container logs with "kubectl logs <pod> --previous" to see the crash reason.
Common causes: application error on startup, missing configuration, failed database connection, missing dependencies.
Check if ConfigMaps or Secrets referenced by the Pod actually exist.
Verify that environment variables are correctly configured.
Check if the container image exists and is accessible from the cluster.

ImagePullBackOff means Kubernetes cannot pull the container image.
Common causes: wrong image name or tag, private registry without credentials, or network issues.
Verify the image name and tag with "kubectl describe pod <pod>" and check the Events section.
For private registries, create a Secret with registry credentials and reference it as imagePullSecrets.

Pending Pods remain in the Pending state because Kubernetes cannot schedule them.
Common causes: insufficient CPU or memory resources, no nodes matching node selectors or affinity rules.
Check with "kubectl describe pod <pod>" and look at the Events section for scheduling failure reasons.
Add more nodes or adjust resource requests to resolve insufficient resources.
Check if PersistentVolumeClaims are in Pending state, which can block Pod scheduling.

Pod is Running but not receiving traffic:
Verify the Service selector matches the Pod labels.
Check the readiness probe: if it fails, the Pod is removed from the Service's endpoint list.
Use "kubectl get endpoints <service>" to verify Pods are listed as endpoints.
Check if NetworkPolicies are blocking traffic between services.

Service cannot be reached from outside the cluster:
Verify the Service type is LoadBalancer or NodePort for external access.
Check the Ingress configuration and ensure the Ingress controller is running.
Verify TLS certificates are valid and correctly configured on the Ingress.
Check cloud provider load balancer health checks and security group rules.

DNS resolution failures in Kubernetes:
CoreDNS is the default DNS service in Kubernetes; check its logs for errors.
Verify the CoreDNS Pods are running with "kubectl get pods -n kube-system".
Test DNS resolution from within a Pod using "nslookup <service>.<namespace>.svc.cluster.local".
Check if the Pod's DNS configuration is correct with "kubectl exec <pod> -- cat /etc/resolv.conf".

Node not ready:
A NotReady node cannot schedule new Pods and existing Pods may be evicted.
Common causes: kubelet not running, insufficient disk space, network issues, or resource exhaustion.
Check node conditions with "kubectl describe node <node>" for details.
Verify kubelet logs on the node for error messages.

Resource quotas and limit ranges:
ResourceQuota limits the total resource consumption in a namespace.
LimitRange sets default and maximum resource limits for containers in a namespace.
Pod creation failures with "exceeded quota" indicate the namespace's resource quota is exhausted.

--------------------------------------------------------------------------------
5.2 Memory Leak Diagnosis
--------------------------------------------------------------------------------

A memory leak occurs when an application allocates memory that is never freed, causing usage to grow over time.
The primary symptom of a memory leak is steadily increasing memory usage over time under constant load.
Memory leaks eventually lead to out-of-memory errors, crashes, or degraded performance.

Common causes of memory leaks in applications:
Unclosed resources: database connections, file handles, HTTP connections, or streams not properly closed.
Growing collections: maps, lists, or caches that accumulate entries without eviction.
Event listeners or callbacks that are registered but never removed.
Closures that capture and retain references to large objects.
Circular references in languages without cycle-detecting garbage collectors.
Global state or static variables that accumulate data.
String concatenation in loops creating many intermediate string objects.

Memory leak diagnosis in Java/JVM:
Use jmap to generate a heap dump: "jmap -dump:format=b,file=heap.bin <pid>".
Analyze heap dumps with Eclipse MAT (Memory Analyzer Tool) or VisualVM.
Look for the largest retained objects and the shortest paths to GC roots.
Use JFR (Java Flight Recorder) for continuous monitoring with minimal overhead.
Enable GC logging to observe garbage collection frequency and memory reclamation.
Frequent full GC cycles with decreasing reclaimed memory indicate a leak.

Memory leak diagnosis in Node.js:
Use the --inspect flag to connect Chrome DevTools for heap profiling.
Take heap snapshots at intervals and compare them to find growing object counts.
Use the heapdump module to generate snapshots programmatically.
The process.memoryUsage() function reports RSS, heap total, heap used, and external memory.
Common Node.js leak sources: unremoved event listeners, unclosed streams, and growing global caches.

Memory leak diagnosis in Python:
The tracemalloc module tracks memory allocations with source file and line number.
Use tracemalloc.take_snapshot() to capture memory state and compare snapshots.
The objgraph module visualizes object reference graphs to find unexpected retention.
gc.get_objects() returns all tracked objects, useful for finding accumulating instances.
Use memory_profiler to decorate functions and track line-by-line memory usage.

General memory leak prevention:
Use connection pools that manage resource lifecycle automatically.
Implement bounded caches with eviction policies (LRU, TTL).
Remove event listeners when they are no longer needed.
Use weak references for caches where entries can be garbage collected if memory is needed.
Monitor memory usage in production with alerts for abnormal growth.
Conduct regular load testing with sustained traffic to surface leaks before production.

--------------------------------------------------------------------------------
5.3 Performance Bottleneck Identification
--------------------------------------------------------------------------------

A performance bottleneck is a point in the system that limits overall throughput or increases latency.
The first step in performance analysis is measuring: profile before optimizing, never guess.
Amdahl's Law states that the maximum speedup from parallelism is limited by the sequential portion of the work.

Application-level profiling:
CPU profiling shows which functions consume the most processing time.
Flame graphs visualize call stacks and time spent, making hot paths immediately visible.
Use language-specific profilers: Java VisualVM, Python cProfile, Node.js clinic.js, Go pprof.
Identify hot loops, excessive object creation, and unnecessary computation.
Measure time spent in I/O operations (database queries, HTTP calls, file access) versus CPU computation.
Most web application bottlenecks are I/O-bound, not CPU-bound.

Database performance:
Slow queries are the most common performance bottleneck in web applications.
Enable slow query logging to capture queries exceeding a threshold (e.g., 100ms).
Use EXPLAIN ANALYZE to examine the query execution plan and identify full table scans.
Missing indexes are the most frequent cause of slow queries.
N+1 query problem: a loop that executes one query per item instead of a single batch query.
Use eager loading or batch fetching to solve N+1 problems.
Connection exhaustion occurs when the application opens more connections than the pool allows.
Monitor active connections, query execution time, and lock contention.

Network-level bottlenecks:
High latency between services in different regions slows synchronous calls.
Use asynchronous processing and message queues to decouple latency-sensitive operations.
DNS resolution can add significant latency; use DNS caching and keep TTLs reasonable.
TLS handshake overhead can be reduced with connection reuse and TLS session resumption.

Caching-related bottlenecks:
Cache miss ratio above 20-30% may indicate the cache is not effective.
Thundering herd: many simultaneous cache misses causing a spike in database load.
Serialize cache rebuilding using distributed locks to prevent thundering herd.

Concurrency bottlenecks:
Lock contention occurs when multiple threads compete for the same lock.
Reduce lock scope to the minimum critical section.
Use read-write locks when reads greatly outnumber writes.
Consider lock-free data structures for high-contention scenarios.
Thread pool exhaustion occurs when all threads are busy and new requests must wait.
Increase thread pool size or reduce per-request processing time.

Load testing for bottleneck discovery:
Gradually increase load to find the point where latency starts increasing.
Monitor all system components (CPU, memory, disk, network, database) during load tests.
Identify the first resource to saturate; that is the bottleneck.
After fixing one bottleneck, retest to find the next one; bottlenecks shift.

--------------------------------------------------------------------------------
5.4 Network Debugging
--------------------------------------------------------------------------------

DNS resolution failure means the system cannot translate a hostname to an IP address.
Use "nslookup" or "dig" to test DNS resolution manually.
Check /etc/resolv.conf for correct DNS server configuration.
DNS caching issues can serve stale records; flush the DNS cache with "systemd-resolve --flush-caches" on Linux.
TTL (Time to Live) determines how long a DNS record is cached; low TTLs enable faster failover.
DNS propagation delays can cause inconsistent resolution across different locations during DNS changes.

SSL/TLS certificate issues:
"Certificate expired" means the TLS certificate's validity period has passed.
"Certificate not trusted" means the certificate is not signed by a trusted Certificate Authority.
"Hostname mismatch" means the certificate's Common Name or Subject Alternative Names do not match the requested domain.
Use "openssl s_client -connect host:port" to inspect a server's certificate details.
Certificate chain issues occur when intermediate certificates are not properly configured.
Check certificate expiration dates and set up automated renewal (e.g., Let's Encrypt with certbot).

Connection timeout vs read timeout:
A connection timeout occurs when the client cannot establish a TCP connection within the allowed time.
A read timeout occurs when the client connects but does not receive a response within the allowed time.
Connection timeouts often indicate network unreachability, firewall rules, or the server not listening.
Read timeouts often indicate slow server processing, deadlocks, or resource exhaustion.

Common network debugging tools:
ping tests basic connectivity and measures round-trip time.
traceroute shows the network path and identifies where delays or failures occur.
curl makes HTTP requests and shows detailed connection and response information.
Use "curl -v" for verbose output including TLS handshake and header details.
netstat or ss shows active connections and listening ports.
tcpdump captures network packets for detailed analysis.
Wireshark provides a graphical interface for analyzing captured network traffic.
Use "nc -zv host port" to test if a specific port is reachable.

Firewall and security group issues:
Incoming traffic may be blocked by host-level firewalls (iptables, ufw) or cloud security groups.
Check both inbound and outbound rules; outbound rules can block responses.
Security groups in cloud environments are stateful; allow the inbound rule and the response is automatically allowed.
Network ACLs in cloud VPCs are stateless; both inbound and outbound rules must be explicitly configured.
Use VPC Flow Logs to diagnose traffic that is being accepted or rejected.

Proxy and load balancer issues:
Verify that the proxy or load balancer is forwarding requests to the correct backend.
Check the X-Forwarded-For header to ensure the original client IP is preserved.
Load balancer health check failures can remove healthy instances from rotation; verify health check configuration.
HTTP/2 and WebSocket connections may require special proxy configuration.

--------------------------------------------------------------------------------
5.5 Database Slow Query Analysis
--------------------------------------------------------------------------------

A slow query is any database query that takes longer than an acceptable threshold to execute.
Enable slow query logging: in MySQL use slow_query_log, in PostgreSQL set log_min_duration_statement.
Identify the slowest queries by execution time and the most frequent slow queries by total accumulated time.
Both individual slow queries and frequently executed moderately slow queries deserve attention.

Using EXPLAIN to analyze queries:
EXPLAIN shows the execution plan the database will use for a query.
EXPLAIN ANALYZE actually runs the query and shows real timing alongside the plan.
Look for sequential scans (Seq Scan) on large tables; these often indicate missing indexes.
A nested loop join on large tables without indexes is a common cause of slow queries.
Hash joins and merge joins are generally more efficient for large dataset joins.

Index optimization:
Create indexes on columns used in WHERE, JOIN, ORDER BY, and GROUP BY clauses.
Composite indexes should order columns from most selective to least selective.
An index on (a, b, c) can support queries filtering on (a), (a, b), or (a, b, c) but not (b) alone.
Use partial indexes to index only relevant rows, reducing index size and maintenance cost.
Remove unused indexes; they slow down writes without benefiting reads.
Use the pg_stat_user_indexes view in PostgreSQL to identify unused indexes.
Covering indexes include all columns needed by a query, avoiding heap table lookups.

Query optimization techniques:
Rewrite subqueries as JOINs when possible; the optimizer handles JOINs more efficiently.
Use EXISTS instead of IN for subqueries when checking for existence.
Avoid SELECT *; specify only the columns you need to reduce data transfer.
Avoid functions on indexed columns in WHERE clauses, as they prevent index usage.
Use LIMIT to restrict results when you do not need all matching rows.
Batch large UPDATE or DELETE operations to avoid long-running transactions and lock contention.
Use prepared statements for frequently executed queries to reduce parse overhead.

Lock contention:
Long-running transactions hold locks that block other queries.
Use "pg_stat_activity" in PostgreSQL to identify blocked queries and blocking processes.
In MySQL, use "SHOW PROCESSLIST" and the information_schema.INNODB_LOCKS tables.
Keep transactions as short as possible; perform computation outside the transaction.
Avoid acquiring locks in inconsistent orders across different code paths, which can cause deadlocks.
Advisory locks provide application-level locking without blocking DDL operations.

Connection management:
Monitor the number of active connections versus the maximum connection limit.
Connection pool exhaustion causes new queries to wait or fail.
Size the connection pool based on the formula: connections = (CPU cores * 2) + disk spindles.
Use PgBouncer or ProxySQL as external connection poolers for better connection management.
Idle connections consume server resources; configure idle timeout to close stale connections.

--------------------------------------------------------------------------------
5.6 Log Analysis Patterns
--------------------------------------------------------------------------------

Effective log analysis identifies patterns, anomalies, and root causes in application behavior.
Structure logs as JSON with consistent fields: timestamp, level, service, request_id, message, and context.
Use correlation IDs to trace a single request across multiple services and log entries.

Common log analysis techniques:
Frequency analysis: count occurrences of log messages to find the most common errors.
Time-series analysis: plot log event counts over time to identify spikes and trends.
Pattern matching: use regular expressions to extract structured data from log messages.
Anomaly detection: establish baseline patterns and alert when deviations occur.
Log aggregation: group related log entries (by request ID, user, or error type) for holistic analysis.

Key patterns to search for:
Error rate spikes: a sudden increase in ERROR-level logs often indicates a deployment issue or upstream failure.
Slow operation warnings: log entries showing operations exceeding expected duration thresholds.
Authentication failures: repeated failed login attempts from the same IP may indicate a brute-force attack.
Resource exhaustion: warnings about connection pools, thread pools, or memory approaching limits.
Timeout patterns: clusters of timeout errors may indicate a downstream service degradation.
Exception stack traces: recurring stack traces point to persistent bugs that need fixing.
Rate limit violations: frequent 429 responses indicate clients that need to adjust their request patterns.

Log-based alerting:
Set up alerts for error rates exceeding a threshold (e.g., more than 10 errors per minute).
Alert on the absence of expected log messages (e.g., no heartbeat logs in 5 minutes).
Use alert deduplication to avoid flooding on-call engineers with the same alert.
Implement alert severity levels: critical alerts page immediately, warnings create tickets.
Include runbook links in alert notifications so responders know how to investigate.

Log retention and management:
Define retention policies based on compliance requirements and debugging needs.
Archive older logs to cold storage (e.g., S3 Glacier) to reduce costs.
Index frequently searched fields for fast query performance.
Sample high-volume debug logs to reduce storage while maintaining visibility.
Redact sensitive data (PII, credentials) from logs before storage.
Use log levels effectively in production: INFO and above by default, with the ability to temporarily enable DEBUG for specific services during investigations.

Incident investigation workflow:
Start with the alert or symptom and identify the time window of the issue.
Search for error logs in the affected time window across all relevant services.
Use the correlation ID to reconstruct the full request path.
Check for recent deployments, configuration changes, or infrastructure events in the same time window.
Examine resource metrics (CPU, memory, connections) alongside logs for correlations.
Document findings in a post-incident report with root cause, impact, timeline, and action items.
Implement automated checks or tests to prevent recurrence of the identified issue.

6.1 Design Patterns
--------------------------------------------------------------------------------

The Singleton pattern ensures a class has only one instance and provides a global point of access to it.
The Factory pattern defines an interface for creating objects, letting subclasses decide which class to instantiate.
The Abstract Factory pattern provides an interface for creating families of related objects without specifying their concrete classes.
The Builder pattern separates the construction of a complex object from its representation.
The Prototype pattern creates new objects by cloning an existing instance.

The Observer pattern defines a one-to-many dependency where all dependents are notified when the subject changes.
The Strategy pattern defines a family of algorithms, encapsulates each one, and makes them interchangeable.
The Command pattern encapsulates a request as an object, allowing parameterization and queuing of requests.
The Decorator pattern attaches additional responsibilities to an object dynamically without altering its structure.
The Adapter pattern converts the interface of a class into another interface that clients expect.
The Facade pattern provides a simplified interface to a complex subsystem.
The Proxy pattern provides a surrogate or placeholder for another object to control access to it.
The Template Method pattern defines the skeleton of an algorithm, deferring some steps to subclasses.
The Iterator pattern provides a way to access elements of a collection sequentially without exposing the underlying representation.
The State pattern allows an object to alter its behavior when its internal state changes.
The Chain of Responsibility pattern passes a request along a chain of handlers until one handles it.
The Mediator pattern defines an object that encapsulates how a set of objects interact.
The Repository pattern abstracts data access behind a collection-like interface.
The Unit of Work pattern maintains a list of objects affected by a business transaction and coordinates writing changes.
The Specification pattern encapsulates a business rule that can be combined with other rules.
The Circuit Breaker pattern (also a design pattern) wraps calls to external services to detect failures and prevent cascading.
The Retry pattern automatically retries failed operations with configurable delay and maximum attempts.
The Saga pattern coordinates a series of local transactions across multiple services.
The Event Sourcing pattern stores the state of an entity as a sequence of state-changing events.
CQRS (Command Query Responsibility Segregation) separates read and write operations into different models.

6.2 Data Structures and Algorithms Fundamentals
--------------------------------------------------------------------------------

An array stores elements in contiguous memory locations, providing O(1) random access by index.
A linked list stores elements in nodes connected by pointers, providing O(1) insertion and deletion but O(n) access.
A stack is a LIFO (Last In, First Out) data structure with push and pop operations, both O(1).
A queue is a FIFO (First In, First Out) data structure with enqueue and dequeue operations, both O(1).
A priority queue returns elements in order of priority, typically implemented with a heap.
A binary heap provides O(log n) insertion and O(log n) extraction of the minimum or maximum element.

A hash table maps keys to values using a hash function, providing average O(1) lookup, insertion, and deletion.
Hash collisions occur when two keys produce the same hash value.
Collision resolution strategies include chaining (linked lists at each bucket) and open addressing (probing).
The load factor of a hash table is the ratio of entries to buckets; resizing occurs when it exceeds a threshold.

A binary search tree (BST) maintains sorted order with O(log n) average search, insert, and delete.
A balanced BST (AVL tree, Red-Black tree) guarantees O(log n) operations by maintaining balance invariants.
A B-tree is a self-balancing tree optimized for disk access, used extensively in database indexes.
A trie (prefix tree) stores strings character by character, enabling efficient prefix searches.

A graph consists of vertices (nodes) and edges (connections between nodes).
Graphs can be directed (edges have direction) or undirected (edges are bidirectional).
Graphs can be weighted (edges have costs) or unweighted.
An adjacency list represents a graph as an array of lists, efficient for sparse graphs.
An adjacency matrix represents a graph as a 2D array, efficient for dense graphs.
Dijkstra's algorithm finds the shortest path in a weighted graph with non-negative edge weights.
Bellman-Ford algorithm handles graphs with negative edge weights.
A topological sort orders vertices in a directed acyclic graph (DAG) such that for every edge u->v, u comes before v.

Binary search finds an element in a sorted array in O(log n) time by repeatedly halving the search space.
Merge sort divides the array in half, sorts each half, and merges them, achieving O(n log n) time complexity.
Quick sort partitions around a pivot and recursively sorts the partitions, with O(n log n) average time.
Counting sort and radix sort achieve O(n) time for integers within a known range.
Dynamic programming solves problems by breaking them into overlapping subproblems and caching results.
Greedy algorithms make the locally optimal choice at each step, sometimes achieving the global optimum.
Big O notation describes the upper bound of an algorithm's time or space complexity as input grows.
O(1) is constant time, O(log n) is logarithmic, O(n) is linear, O(n log n) is linearithmic, O(n^2) is quadratic.

--------------------------------------------------------------------------------
6.3 Operating System Concepts
--------------------------------------------------------------------------------

A process is an instance of a running program with its own memory space, file descriptors, and state.
A thread is a lightweight unit of execution within a process, sharing the process's memory space.
Context switching is the overhead of saving and restoring the state when switching between processes or threads.
Virtual memory maps logical addresses to physical addresses, allowing processes to use more memory than physically available.
Paging divides memory into fixed-size blocks (pages) and manages them through a page table.
A page fault occurs when a program accesses a page not currently in physical memory, triggering disk I/O.
Thrashing occurs when excessive page faults cause the system to spend more time swapping than executing.

A file descriptor is a handle that a process uses to access files, sockets, and other I/O resources.
Standard input (stdin), standard output (stdout), and standard error (stderr) are file descriptors 0, 1, and 2.
A pipe connects the stdout of one process to the stdin of another for inter-process communication.
A socket is an endpoint for network communication, identified by an IP address and port number.
TCP (Transmission Control Protocol) provides reliable, ordered delivery of data over a network.
UDP (User Datagram Protocol) provides fast, connectionless delivery without guarantees.
The TCP three-way handshake establishes a connection: SYN, SYN-ACK, ACK.
TCP uses flow control (sliding window) and congestion control to manage data transmission rates.
The ephemeral port range (typically 49152-65535) is used for client-side connections.
Port exhaustion occurs when all ephemeral ports are in use, preventing new outbound connections.

A mutex (mutual exclusion) prevents multiple threads from simultaneously accessing a shared resource.
A semaphore controls access to a shared resource by maintaining a count of available permits.
A deadlock occurs when two or more threads are each waiting for a resource held by the other.
The four conditions for deadlock are: mutual exclusion, hold and wait, no preemption, and circular wait.
Breaking any one of the four conditions prevents deadlock.

Linux namespaces isolate system resources (PID, network, mount, user) for containers.
Linux cgroups (control groups) limit and account for resource usage (CPU, memory, I/O) of process groups.
Containers use namespaces for isolation and cgroups for resource limits.
The OOM (Out of Memory) killer terminates processes when the system runs out of memory.
The /proc filesystem provides information about running processes and system configuration.
The /sys filesystem exposes kernel device and driver information.
System calls are the interface between user-space programs and the kernel.
The select, poll, and epoll system calls enable efficient I/O multiplexing for handling many concurrent connections.
epoll is the most efficient for large numbers of connections on Linux, used by NGINX and Node.js.

--------------------------------------------------------------------------------
6.4 Networking Fundamentals
--------------------------------------------------------------------------------

The OSI model divides network communication into seven layers: Physical, Data Link, Network, Transport, Session, Presentation, Application.
The TCP/IP model uses four layers: Network Access, Internet, Transport, and Application.
An IP address uniquely identifies a device on a network; IPv4 uses 32 bits, IPv6 uses 128 bits.
A subnet mask defines the network and host portions of an IP address.
CIDR notation (e.g., 192.168.1.0/24) specifies an IP address range with a prefix length.
A MAC address is a hardware identifier assigned to a network interface, used at the Data Link layer.
ARP (Address Resolution Protocol) maps IP addresses to MAC addresses on a local network.
NAT (Network Address Translation) allows multiple devices to share a single public IP address.

DNS (Domain Name System) translates human-readable domain names to IP addresses.
A DNS A record maps a domain name to an IPv4 address.
A DNS AAAA record maps a domain name to an IPv6 address.
A DNS CNAME record creates an alias pointing to another domain name.
A DNS MX record specifies the mail server for a domain.
A DNS TXT record stores arbitrary text, often used for SPF, DKIM, and domain verification.
DNS resolution follows a hierarchy: local cache, recursive resolver, root server, TLD server, authoritative server.
DNS TTL (Time to Live) specifies how long a record should be cached before re-querying.

A VLAN (Virtual Local Area Network) logically segments a physical network into isolated broadcast domains.
A VPN (Virtual Private Network) creates an encrypted tunnel between networks over the public internet.
A firewall filters network traffic based on rules specifying allowed and denied connections.
A WAF (Web Application Firewall) protects web applications by filtering and monitoring HTTP traffic.
A DDoS (Distributed Denial of Service) attack floods a target with traffic from multiple sources.
DDoS mitigation services like Cloudflare and AWS Shield absorb and filter attack traffic.
A reverse proxy sits in front of web servers, handling client requests and forwarding them to backends.
mTLS (mutual TLS) requires both client and server to present certificates, providing two-way authentication.

HTTP/1.1 uses text-based headers and allows one request per TCP connection (with keep-alive for reuse).
HTTP/2 uses binary framing, multiplexes multiple streams over a single TCP connection, and supports header compression.
HTTP/3 uses QUIC (based on UDP) instead of TCP, reducing connection establishment latency.
gRPC is a high-performance RPC framework that uses HTTP/2 and Protocol Buffers for efficient binary serialization.
Protocol Buffers (protobuf) is a language-neutral, platform-neutral serialization format more efficient than JSON.
WebSocket provides full-duplex communication over a single TCP connection after an HTTP upgrade handshake.
Server-Sent Events (SSE) provide one-way server-to-client streaming over HTTP.

6.5 Practical Linux Commands for Troubleshooting
--------------------------------------------------------------------------------

The "top" and "htop" commands show real-time system resource usage including CPU, memory, and processes.
The "ps aux" command lists all running processes with their PID, CPU, memory usage, and command.
The "df -h" command shows disk space usage for all mounted filesystems in human-readable format.
The "du -sh /path" command shows the total disk usage of a specific directory.
The "free -m" command displays total, used, and available memory in megabytes.
The "uptime" command shows how long the system has been running and the load averages.
Load average represents the average number of processes in the runnable or waiting state over 1, 5, and 15 minutes.
A load average higher than the number of CPU cores indicates the system is overloaded.

The "lsof -i :port" command lists processes using a specific network port.
The "ss -tulnp" command shows all listening TCP and UDP sockets with their owning processes.
The "iptables -L" command lists all firewall rules on a Linux system.
The "journalctl -u service" command views logs for a specific systemd service.
The "dmesg" command shows kernel ring buffer messages, useful for hardware and driver issues.
The "strace -p PID" command traces system calls made by a running process.
The "lsblk" command lists all block devices and their mount points.
The "iostat" command reports CPU and I/O statistics for devices and partitions.
The "vmstat" command reports virtual memory statistics including processes, memory, paging, and CPU activity.
The "sar" command collects, reports, and saves system activity information for historical analysis.
The "nmap host" command scans a host for open ports and running services.
The "curl -w @- <<< format" command with format strings provides detailed timing for HTTP requests.

The "chmod" command changes file permissions using numeric (644) or symbolic (u+x) notation.
The "chown" command changes the owner and group of a file or directory.
The "find / -name filename" command searches for files by name recursively from the specified directory.
The "grep -r pattern /path" command searches for a text pattern recursively within files.
The "tail -f /var/log/syslog" command follows a log file in real time, showing new entries as they are appended.
The "awk" command processes text files line by line, useful for extracting and transforming structured data.
The "sed" command performs text transformations on streams or files, commonly used for search and replace.
The "xargs" command builds and executes commands from standard input, useful for batch operations.
The "watch -n 5 command" command repeatedly executes a command every 5 seconds, displaying the output.
The "tmux" or "screen" commands provide terminal multiplexing, allowing persistent sessions that survive disconnection.

6.6 API Design Best Practices
--------------------------------------------------------------------------------

Use nouns, not verbs, in REST endpoint URLs: /users not /getUsers.
Use plural nouns for collection endpoints: /products, /orders, /users.
Nest resources to express relationships: /users/{id}/orders lists orders for a specific user.
Keep URLs lowercase and use hyphens to separate words: /user-profiles not /userProfiles.
Use query parameters for filtering, sorting, and pagination: /products?category=electronics&sort=price.
Return consistent response envelopes with data, metadata, and error fields.
Use ISO 8601 format for dates and times in API responses: 2025-03-24T14:30:00Z.
Support content negotiation by respecting the Accept header.
Version your API from the start to allow backward-incompatible changes in future versions.
Provide idempotency keys for POST endpoints to allow safe retries.
Document every endpoint, including request/response examples, error codes, and rate limits.
Use HTTP 202 Accepted for long-running operations and provide a status endpoint for polling.
Return meaningful error responses with a machine-readable error code and a human-readable message.
Implement request validation early in the pipeline and return 400 with specific field-level errors.
Use ETags and conditional requests (If-None-Match) to reduce bandwidth for unchanged resources.
Support partial responses using a fields query parameter to let clients request only needed data.
Implement bulk operations (e.g., POST /users/batch) for clients that need to create or update many resources.
Use hypermedia links (HATEOAS) to make the API self-discoverable.
Log every API request with a unique request ID for tracing and debugging.
Monitor API usage per client, per endpoint, and per response status for operational visibility.

6.7 Container Security
--------------------------------------------------------------------------------

Run containers as a non-root user to limit the impact of container breakouts.
Use minimal base images (distroless, Alpine) to reduce the attack surface.
Scan container images for known vulnerabilities using tools like Trivy, Grype, or Snyk.
Never store secrets (passwords, API keys) in container images; use Secret management tools instead.
Use read-only filesystems for containers when possible to prevent runtime modifications.
Sign container images and verify signatures before deployment to prevent tampering.
Implement network policies in Kubernetes to restrict which pods can communicate with each other.
Use Pod Security Standards (Restricted, Baseline, Privileged) to enforce security constraints.
Drop all Linux capabilities and add back only the ones the container needs.
Set resource limits on every container to prevent resource abuse and denial-of-service.
Regularly update base images to incorporate security patches.
Use multi-stage builds to ensure build tools and source code are not included in the final image.
Enable audit logging for container orchestration platforms to track administrative actions.
Implement runtime security monitoring with tools like Falco to detect anomalous container behavior.
Use image pull policies of "Always" in production to ensure the latest patched images are used.
Store container images in private registries with access controls and vulnerability scanning.

--------------------------------------------------------------------------------
6.8 Observability Principles
--------------------------------------------------------------------------------

Observability is the ability to understand the internal state of a system from its external outputs.
The three pillars of observability are metrics, logs, and traces.
Metrics are numerical measurements collected at regular intervals, useful for alerting and dashboards.
Logs are timestamped records of discrete events, useful for debugging specific issues.
Traces track the lifecycle of a request across distributed services.

OpenTelemetry is an open standard for collecting metrics, logs, and traces across applications.
OpenTelemetry provides SDKs for most programming languages to instrument applications.
A span represents a unit of work in a trace, such as an HTTP request or a database query.
A trace is a collection of spans forming a directed acyclic graph representing a request's journey.
Context propagation passes trace context between services via HTTP headers or message metadata.
The W3C Trace Context standard defines the traceparent and tracestate headers for context propagation.

Structured events combine the richness of logs with the queryability of metrics.
High-cardinality fields (like user ID or request ID) enable powerful filtering but are expensive to index.
Sampling reduces the volume of collected traces while preserving visibility into system behavior.
Head-based sampling decides whether to trace a request at the start, before any spans are created.
Tail-based sampling makes the decision after the trace is complete, keeping interesting traces (errors, slow requests).
Exemplars link metrics to specific traces, allowing you to drill down from a dashboard into a specific request.

Service Level Indicators (SLIs) should be defined from the user's perspective.
Request latency at the 50th, 95th, and 99th percentiles captures the full user experience.
Error rate as a percentage of total requests is a fundamental reliability indicator.
Availability measured as successful requests divided by total requests is the most common SLI.
Set error budgets based on SLOs to balance reliability with development velocity.
When the error budget is exhausted, prioritize reliability work over new features.
Runbooks should be linked to every alert, providing step-by-step investigation and remediation guidance.
Dashboards should answer specific questions rather than displaying every available metric.
Use the RED method (Rate, Errors, Duration) for monitoring request-driven services.
Use the USE method (Utilization, Saturation, Errors) for monitoring infrastructure resources.
