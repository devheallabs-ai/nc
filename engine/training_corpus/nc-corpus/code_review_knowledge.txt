NOVA CODE REVIEW KNOWLEDGE BASE

################################################################################
SECTION 1: CODE REVIEW FUNDAMENTALS
################################################################################

--------------------------------------------------------------------------------
1.1 What is Code Review
--------------------------------------------------------------------------------

Code review is the systematic examination of source code to find bugs, security issues, and quality problems.
A code review checks for correctness, security, performance, readability, and maintainability.
Automated code review uses static analysis to detect patterns that indicate bugs or vulnerabilities.
Code review improves software quality by catching defects before they reach production.
A code review score measures the overall health of a codebase on a scale from 0 to 100.
Code quality is measured by the ratio of clean files to total files in a project.
A clean file has no security issues, no anti-patterns, and follows language conventions.
A code review issue is a problem that must be fixed, such as a security vulnerability.
A code review warning is a suggestion for improvement that is not critical.
Code health score is calculated as 100 multiplied by clean files divided by total files.

Code review detects security vulnerabilities before deployment.
Code review identifies anti-patterns that reduce maintainability.
Code review enforces consistent coding standards across a team.
Code review catches performance problems early in development.
Code review finds hardcoded secrets and credentials in source code.

--------------------------------------------------------------------------------
1.2 Security Vulnerabilities (All Languages)
--------------------------------------------------------------------------------

A hardcoded password is a security vulnerability where credentials are stored in source code.
Hardcoded passwords should be replaced with environment variables or secret managers.
A hardcoded API key in source code can be exploited if the repository is public.
Secret detection finds passwords, tokens, and API keys embedded in code.
Never commit secrets to version control because they persist in git history forever.

SQL injection occurs when user input is concatenated directly into SQL queries.
SQL injection is prevented by using parameterized queries or prepared statements.
String concatenation in SQL queries creates SQL injection vulnerabilities.
Always use parameterized queries to prevent SQL injection attacks.

Cross-site scripting XSS occurs when untrusted data is rendered as HTML without escaping.
innerHTML is vulnerable to XSS because it renders raw HTML from untrusted input.
Use textContent instead of innerHTML to prevent XSS attacks.
Input sanitization removes or escapes dangerous characters from user input.
Output encoding converts special characters to safe HTML entities before rendering.

Command injection occurs when user input is passed to shell commands without sanitization.
The system function in C executes shell commands and is vulnerable to command injection.
The eval function in Python and JavaScript executes arbitrary code and is a security risk.
The exec function in Python executes arbitrary code strings and should never use untrusted input.
The pickle.load function in Python can execute arbitrary code from untrusted data.
Deserialization of untrusted data is a security vulnerability in Java, Python, and PHP.

Path traversal occurs when user input manipulates file paths to access unauthorized files.
Directory traversal uses sequences like ../ to escape the intended directory.
Always validate and sanitize file paths to prevent path traversal attacks.

Buffer overflow occurs when data exceeds the allocated memory boundary in C and C++.
The gets function in C has no bounds checking and causes buffer overflow vulnerabilities.
Use fgets instead of gets to prevent buffer overflow in C programs.
The strcpy function copies without bounds checking and can cause buffer overflow.
Use strncpy or strlcpy instead of strcpy for safe string copying.
The sprintf function has no bounds checking and should be replaced with snprintf.

--------------------------------------------------------------------------------
1.3 Universal Code Quality Checks
--------------------------------------------------------------------------------

TODO comments indicate incomplete work that should be tracked and resolved.
FIXME comments indicate known bugs that need to be fixed.
HACK comments indicate temporary workarounds that should be replaced with proper solutions.
Dead code is unreachable code that should be removed to improve maintainability.
Commented-out code should be removed because version control preserves history.

Files larger than 500 lines should be considered for splitting into smaller modules.
Functions longer than 50 lines should be refactored into smaller, focused functions.
Deeply nested code with more than 3 levels of indentation reduces readability.
Cyclomatic complexity measures the number of independent paths through code.
High cyclomatic complexity indicates code that is difficult to test and maintain.

Magic numbers are unexplained numeric literals that should be replaced with named constants.
Duplicate code should be refactored into shared functions or modules.
Global variables create hidden dependencies and make code harder to test.
Mutable global state is a source of race conditions in concurrent programs.

SECTION 2: PYTHON CODE REVIEW
################################################################################

--------------------------------------------------------------------------------
2.1 Python Security
--------------------------------------------------------------------------------

Python eval executes arbitrary code and is a critical security vulnerability.
Python exec executes arbitrary code strings and should never process user input.
Python pickle.load deserializes data and can execute arbitrary code from untrusted sources.
Python yaml.load without SafeLoader can execute arbitrary code from YAML files.
Use yaml.safe_load instead of yaml.load to prevent code execution from YAML.
Python subprocess.shell=True is vulnerable to command injection.
Use subprocess with a list of arguments instead of shell=True for safety.
Python os.system executes shell commands and is vulnerable to command injection.
Use subprocess.run instead of os.system for safer command execution.

Python input function in Python 2 evaluates expressions and is a security risk.
Python assert statements are removed when code runs with optimization flags.
Never use assert for security checks because optimization removes assertions.

--------------------------------------------------------------------------------
2.2 Python Best Practices
--------------------------------------------------------------------------------

Python bare except catches all exceptions including SystemExit and KeyboardInterrupt.
Use except Exception instead of bare except to avoid catching system exits.
Python wildcard import with import star pollutes the namespace and hides dependencies.
Always use explicit imports to make dependencies clear and avoid name conflicts.
Python missing __name__ guard allows module code to run during import.
Add if __name__ == "__main__" guard to Python scripts that should not run on import.

Python mutable default arguments like def f(x=[]) share state between calls.
Use None as default and create new objects inside the function body.
Python string concatenation in loops is O(n^2) and should use join instead.
Python global keyword creates hidden dependencies between functions.
Python nested try-except blocks reduce readability and should be flattened.

Type hints improve Python code readability and enable static type checking.
Python docstrings document functions, classes, and modules for users and tools.
Python f-strings are preferred over format() and % formatting for readability.
Python context managers with statement ensure proper resource cleanup.
Python list comprehensions are more readable and faster than equivalent for loops.
Python dataclasses reduce boilerplate for classes that primarily store data.

--------------------------------------------------------------------------------
2.3 Python Performance
--------------------------------------------------------------------------------

Python dictionary lookups are O(1) and faster than searching lists which are O(n).
Python generators use lazy evaluation and save memory for large datasets.
Python collections.Counter efficiently counts elements without manual loops.
Python functools.lru_cache memoizes function results to avoid redundant computation.
Python set operations like intersection and union are faster than list-based equivalents.
Python numpy vectorized operations are orders of magnitude faster than Python loops.

SECTION 3: JAVA CODE REVIEW
################################################################################

--------------------------------------------------------------------------------
3.1 Java Security
--------------------------------------------------------------------------------

Java SQL injection occurs when queries are built with string concatenation.
Use PreparedStatement instead of Statement to prevent SQL injection in Java.
Java deserialization of untrusted data using ObjectInputStream can execute arbitrary code.
Java Runtime.exec with unsanitized input is vulnerable to command injection.
Java reflection allows bypassing access controls and should be used carefully.
Java weak random from java.util.Random is predictable and should not be used for security.
Use java.security.SecureRandom for cryptographic random number generation.

Java XML external entity XXE attacks exploit XML parsers that process external entities.
Disable external entity processing in Java XML parsers to prevent XXE attacks.
Java Server-Side Request Forgery SSRF occurs when user input controls outgoing HTTP requests.

--------------------------------------------------------------------------------
3.2 Java Best Practices
--------------------------------------------------------------------------------

Java System.out.println should be replaced with a proper logging framework.
Use SLF4J or Log4j2 instead of System.out for production Java logging.
Java catching broad Exception hides specific error types and makes debugging harder.
Catch specific exceptions to handle each error case appropriately.
Java raw types lose type safety and should use generics instead.
Java new Thread should be replaced with ExecutorService for managed thread pools.
Java Thread.sleep in production code blocks threads and wastes resources.

Java Optional reduces null pointer exceptions by making absence explicit.
Java Stream API provides declarative data processing over collections.
Java try-with-resources ensures proper resource cleanup for AutoCloseable objects.
Java immutable objects are thread-safe and prevent unintended state changes.
Java final fields should be used for values that do not change after construction.
Java diamond operator <> reduces verbosity in generic type instantiation.

--------------------------------------------------------------------------------
3.3 Java Performance
--------------------------------------------------------------------------------

Java StringBuilder is faster than string concatenation in loops.
Java HashMap provides O(1) average lookup time for key-value access.
Java ConcurrentHashMap is thread-safe and faster than synchronizing HashMap.
Java lazy initialization defers expensive object creation until first use.
Java connection pooling reuses database connections to reduce overhead.
Java garbage collection tuning improves latency for memory-intensive applications.


################################################################################
SECTION 4: JAVASCRIPT AND TYPESCRIPT CODE REVIEW
################################################################################

--------------------------------------------------------------------------------
4.1 JavaScript Security
--------------------------------------------------------------------------------

JavaScript eval executes arbitrary code and is a critical security vulnerability.
JavaScript innerHTML renders raw HTML and is vulnerable to XSS attacks.
Use textContent or DOM methods instead of innerHTML to prevent XSS.
JavaScript document.write injects content into the page and is vulnerable to XSS.
JavaScript window.location with user input can redirect users to malicious sites.
JavaScript postMessage without origin validation allows cross-origin attacks.

Node.js child_process.exec is vulnerable to command injection with user input.
Use child_process.execFile with arguments array for safer command execution.
Node.js require with user-controlled paths enables arbitrary code loading.
Node.js path traversal via user input in fs operations exposes system files.

--------------------------------------------------------------------------------
4.2 JavaScript Best Practices
--------------------------------------------------------------------------------

JavaScript var has function scope and hoisting, use let and const instead.
JavaScript let provides block scoping and should replace var in modern code.
JavaScript const prevents reassignment and signals immutable bindings.
JavaScript console.log should be removed from production code.
JavaScript callback hell creates deeply nested code that is hard to read.
Use promises or async/await instead of nested callbacks.
JavaScript == performs type coercion, use === for strict equality comparison.

TypeScript any type disables type checking and should be avoided.
TypeScript unknown is safer than any because it requires type narrowing before use.
TypeScript interfaces define object shapes and enable structural typing.
TypeScript strict mode enables all strict type checking options.
TypeScript enum can cause code bloat and should use const enum or union types.

JavaScript arrow functions provide concise syntax and lexical this binding.
JavaScript destructuring extracts values from objects and arrays cleanly.
JavaScript template literals enable multi-line strings and expression interpolation.
JavaScript optional chaining with ?. safely accesses nested properties.
JavaScript nullish coalescing with ?? provides defaults only for null and undefined.

--------------------------------------------------------------------------------
4.3 JavaScript Performance
--------------------------------------------------------------------------------

JavaScript DOM manipulation should be batched to reduce layout reflows.
JavaScript event delegation attaches one handler to a parent instead of many to children.
JavaScript lazy loading defers loading non-critical resources until needed.
JavaScript web workers offload CPU-intensive tasks to background threads.
JavaScript debouncing limits function execution frequency for scroll and resize events.
JavaScript memoization caches function results to avoid redundant computation.

SECTION 5: GO CODE REVIEW
################################################################################

--------------------------------------------------------------------------------
5.1 Go Best Practices
--------------------------------------------------------------------------------

Go error handling should check every error return value explicitly.
Go ignored errors with _ = err hide failures and cause silent bugs.
Go panic should not be used in library code because it terminates the program.
Use error returns instead of panic for recoverable failures in Go.
Go fmt.Println should be replaced with structured logging in production.
Use log/slog or zerolog for structured Go logging.
Go goroutine leaks occur when goroutines block indefinitely without cleanup.
Use context.Context for goroutine cancellation and timeout management.

Go interfaces should be small, typically one to three methods.
Go struct embedding provides composition without inheritance.
Go defer ensures cleanup runs even if a function panics.
Go channels enable safe communication between goroutines.
Go sync.Mutex protects shared state from concurrent access.
Go race detector with go test -race detects data races in concurrent code.

--------------------------------------------------------------------------------
5.2 Go Security
--------------------------------------------------------------------------------

Go SQL injection occurs when queries use fmt.Sprintf with user input.
Use database/sql with parameterized queries to prevent SQL injection in Go.
Go template injection occurs when user input is used in template construction.
Go path traversal via filepath.Join with user input can escape the base directory.
Use filepath.Clean and validate paths are within allowed directories.
Go TLS configuration should enforce minimum TLS 1.2 and strong cipher suites.
Go crypto/rand should be used instead of math/rand for security-sensitive random values.

SECTION 6: C AND C++ CODE REVIEW
################################################################################

--------------------------------------------------------------------------------
6.1 C Security
--------------------------------------------------------------------------------

C gets function has no bounds checking and is removed from C11 standard.
Always use fgets instead of gets to prevent buffer overflow in C.
C strcpy copies without bounds checking and causes buffer overflow.
Use strncpy or strlcpy for safe string copying in C.
C sprintf has no bounds checking and should be replaced with snprintf.
C strcat concatenates without bounds checking and can overflow the destination.
C scanf with %s has no length limit and causes buffer overflow.

C malloc without checking the return value causes null pointer dereference.
Always check malloc return value for NULL before using the allocated memory.
C free on already freed memory causes use-after-free vulnerability.
C double free corrupts heap metadata and can be exploited for code execution.
C integer overflow wraps around silently and causes unexpected behavior.
C format string vulnerability occurs when user input is passed as printf format.
Always use printf("%s", input) instead of printf(input) to prevent format string attacks.

C system function executes shell commands and is vulnerable to command injection.
Use exec family functions instead of system for safer process creation.
C race condition in file operations between check and use is called TOCTOU.
Use atomic file operations to prevent TOCTOU race conditions.

--------------------------------------------------------------------------------
6.2 C Best Practices
--------------------------------------------------------------------------------

C functions should have single responsibility and be under 50 lines.
C header guards prevent multiple inclusion with ifndef define endif.
C const correctness marks read-only parameters and prevents accidental modification.
C static functions limit visibility to the current translation unit.
C enum provides named constants instead of magic numbers.
C struct padding affects memory layout and should be considered for performance.
C volatile keyword prevents compiler optimization of memory-mapped hardware registers.

C memory leaks occur when allocated memory is never freed.
Use tools like Valgrind and AddressSanitizer to detect memory leaks.
C undefined behavior includes signed integer overflow, null dereference, and out-of-bounds access.
C compiler warnings should be treated as errors with -Werror flag.
C static analysis tools like cppcheck and clang-analyzer find bugs before runtime.

SECTION 7: NC LANGUAGE CODE REVIEW
################################################################################

--------------------------------------------------------------------------------
7.1 NC Conventions
--------------------------------------------------------------------------------

NC services must declare a service name with service keyword.
NC services should declare a version with version keyword.
NC uses to keyword for defining behaviors, not function keyword.
NC uses respond with instead of return for producing output from behaviors.
NC configure block sets up service initialization and default settings.
NC middleware block adds request processing middleware.
NC health endpoint provides service health checking for monitoring.
NC log statement provides structured logging within services.

NC import loads modules from the module path.
NC set assigns values to variables.
NC show prints output to the console.
NC repeat-while creates loops in NC.
NC if-else provides conditional branching in NC.
NC list creates ordered collections in NC.
NC dict creates key-value mappings in NC.

--------------------------------------------------------------------------------
7.2 NC Anti-patterns
--------------------------------------------------------------------------------

NC function keyword is not standard, use to keyword for behavior definitions.
NC return keyword is not standard, use respond with for output.
NC missing service declaration means the file is not a proper NC module.
NC missing version means the service cannot be tracked across releases.
NC missing health endpoint means monitoring tools cannot check service status.
NC missing configure block means defaults are not explicitly set.
NC deeply nested loops beyond 2 levels cause VM segfaults.
NC long behavior chains with many function calls can cause stack overflow.

SECTION 8: RUST CODE REVIEW
################################################################################

--------------------------------------------------------------------------------
8.1 Rust Best Practices
--------------------------------------------------------------------------------

Rust unwrap panics on error and should be replaced with proper error handling.
Use the question mark operator for propagating errors in Rust.
Rust clone should be avoided when borrowing is possible.
Rust unsafe blocks bypass the borrow checker and should be minimized.
Rust lifetime annotations ensure references are valid for their required scope.
Rust clippy provides automated linting with hundreds of lint rules.
Rust cargo fmt enforces consistent code formatting across a project.

SECTION 9: CODE REVIEW SCORING
################################################################################

--------------------------------------------------------------------------------
9.1 Scoring Model
--------------------------------------------------------------------------------

Code review scoring assigns points based on the severity of detected issues.
A critical security issue such as SQL injection or eval reduces the score by 20 points.
A moderate security issue such as hardcoded password reduces the score by 15 points.
A best practice violation reduces the score by 5 to 10 points.
An informational suggestion reduces the score by 2 to 3 points.
The minimum code review score is 0 and the maximum is 100.
A score above 80 indicates good code quality.
A score between 60 and 80 indicates acceptable code quality with room for improvement.
A score below 60 indicates code that needs significant improvement.
A score below 40 indicates code with critical issues that must be addressed.

Code review categories include security, performance, maintainability, and correctness.
Security issues have the highest priority in code review.
Performance issues affect user experience and resource costs.
Maintainability issues increase the cost of future changes.
Correctness issues cause bugs and unexpected behavior.

--------------------------------------------------------------------------------
9.2 Review Recommendations
--------------------------------------------------------------------------------

Code review should prioritize security vulnerabilities first.
Code review should then address correctness and logic errors.
Code review should next focus on performance bottlenecks.
Code review should finally suggest maintainability improvements.
Code review recommendations should include the specific file and issue location.
Code review recommendations should explain why the issue matters.
Code review recommendations should suggest a concrete fix.
Code review should provide a health score summarizing overall code quality.
Code review should list the number of files scanned, issues found, and warnings raised.
Code review should categorize findings as issues, warnings, or suggestions.

SECTION 10: LANGUAGE-SPECIFIC REVIEW PATTERNS
################################################################################

--------------------------------------------------------------------------------
10.1 Pattern Detection
--------------------------------------------------------------------------------

Pattern detection identifies common code patterns that indicate potential problems.
Static analysis checks code without executing it to find bugs and vulnerabilities.
Linting enforces coding standards and catches common mistakes.
Code smell is a surface indication that usually corresponds to a deeper problem.
Anti-pattern is a common response to a recurring problem that is counterproductive.

Python code review checks for eval, exec, wildcard imports, bare except, and missing guards.
Java code review checks for SQL injection, System.out, broad exception catches, and raw threads.
JavaScript code review checks for eval, innerHTML, var usage, console.log, and any types.
Go code review checks for ignored errors, panic in libraries, fmt.Println, and race conditions.
C code review checks for gets, strcpy, sprintf, system, buffer overflow, and memory leaks.
NC code review checks for missing service, missing version, function keyword, and return keyword.
Rust code review checks for unwrap, unnecessary clone, unsafe blocks, and missing error handling.

Multi-language code review applies universal checks across all programming languages.
Universal checks include hardcoded secrets, TODO markers, file size limits, and dead code.
Language-specific checks apply rules that are relevant to each programming language.
A comprehensive code review combines universal and language-specific checks.
